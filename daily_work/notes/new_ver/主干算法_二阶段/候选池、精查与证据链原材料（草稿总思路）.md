## 0. 文档定位

本文不是具体模型选型文档，也不是 detector 具体算法实现文档。本文只确定下一阶段的**数据流动链路、模块边界、候选池来源、四类业务 detector 接口、survival 精查位置、证据缓存与线索卡原材料输出方向**。

当前已经完成的工作是主干概率 scorer 的 v1 验证：

```text
probability_candidate_v1 = logistic_regression + frequency_decay_v1 + raw
business_usable_probability_baseline = true
```

它可以作为全量 entity 的初筛 scorer，但它不是完整主干算法的终点。

完整下一阶段应从“全量概率预测”升级为：

```text
全量风险 scorer
→ 业务候选池生成
→ TopK survival / interval-aware 精查
→ TopK detector 证据补充
→ structured evidence bundle
→ 后续 LLM / MCP 线索卡生成
```

## 1. 需求侧 detector 分类

本阶段必须回到需求文档的四大类，而不是直接按算法 detector 名称组织。

### 1.1 价格预警类

业务目标：识别可能影响药企价格体系的异常低价或跨区域价差。

规则来源：

```text
1. 低价采购预警
2. 订单价差异常
```

业务作用：

```text
发现低于客户预警价的采购行为；
发现同品规折算至单位可比价后的异常价差；
提示人工核实价格来源、预警价维护情况、配送企业、医院情况、跨省价格差异。
```

当前算法阶段定位：

```text
v1 定义接口，暂不强依赖当前脱敏采购价实现。
```

原因是当前 `purchase_price` 可能被独立脱敏，不适合直接做真实价格判断。价格类 detector 应作为未来接入真实价格/可比价口径后的 detector family。

### 1.2 配送响应类

业务目标：识别医院已经下单，但配送企业响应失败、响应慢或配送率不足。

规则来源：

```text
1. 拒绝响应预警
2. 响应不及时预警
3. 配送率低预警
```

业务作用：

```text
识别拒绝配送、无法配送、缺货、未及时配送、拒收、退货等异常状态；
识别确认订单后超过规定时间仍未发货；
识别配送数量 / 采购数量低于阈值的情况；
帮助判断风险是否来自配送企业，而不是医院需求流失。
```

当前算法阶段定位：

```text
v1 应优先设计并可部分实现。
```

配送响应类与终端不丢高度相关，因为它能区分：

```text
医院不想买了
vs
医院想买但供不上
```

这类证据应进入 TopK 候选的精查阶段。

### 1.3 丢失新进类 / 终端动态类

业务目标：识别医院-药品关系的长期未采购和新进入关系。

规则来源：

```text
1. 终端丢失预警
2. 新进终端识别
```

业务作用：

```text
根据历史采购数量、采购周期、预计库存消耗周期判断是否超期未采购；
识别此前没有采购或丢失超过 180 天后重新采购的终端；
判断新进终端是否具有有效采购量；
辅助业务员评估是否需要挽回、跟进或观察。
```

当前算法阶段定位：

```text
这是当前主干概率 scorer + survival 精查的核心业务场景。
```

当前 `logistic_regression + frequency_decay_v1 + raw` 属于这一类的 Stage 1 初筛 scorer。后续 survival / interval-aware 精查也首先服务于这一类。

### 1.4 销量波动类

业务目标：识别采购数量或采购频次相对于历史水平的异常上升或下降。

规则来源：

```text
1. 采购量异常波动
2. 采购频次异常波动
```

业务作用：

```text
识别采购量超过近 6 个月平均采购量多倍，或与上月相比骤降；
识别近 30 天采购次数超过近 6 个月平均频次多倍，或与上月相比骤降；
辅助判断囤货、断货、季节性需求、采购模式异常；
与终端丢失、新进终端联动分析。
```

当前算法阶段定位：

```text
v1 应作为 TopK 候选的证据型 detector。
```

当前 `frequency_decay_v1` 已经吸收了一部分“采购频次下降”的思想，但它仍属于 scorer 特征，不等价于完整销量波动 detector。后续需要输出结构化证据：例如近期频次是否显著低于历史频次、是否连续多 cutoff 恶化。

## 2. 当前主干模型的重新定位

当前完成的概率模型不是完整业务排序模型，而是：

```text
Stage 1 full-entity pointwise risk scorer
```

它的职责是：

```text
对全量 recurring entity × cutoff × horizon 输出 churn_probability_H；
提供初步概率排序；
支持生成概率 TopK 候选；
支持后续 business priority 计算。
```

它不负责：

```text
最终业务优先级；
one-shot 高价值召回；
survival 风险状态；
detector 证据；
根因归因；
线索卡自然语言生成；
自动派单。
```

因此下一阶段不是继续“模型调优”，而是设计候选池和精查流水线。

## 3. 总体数据流

本阶段主线数据流应为：

```text
data/03_cleaned/bs_agent_dingdan_model_base.parquet
        │
        ▼
fact_purchase_event / entity_month / demand_profile / cutoff features
        │
        ▼
Stage 1: full-entity risk scorer
        │
        ├── churn_probability_H
        ├── probability_rank
        └── risk_band
        │
        ▼
Stage 1.5: candidate pool generation
        │
        ├── probability_top5pct candidates
        ├── business_priority_top5pct candidates
        ├── one_shot_high_value_attention candidates
        └── demand_shape_observation candidates
        │
        ▼
Stage 2: survival / interval-aware refinement on candidate pool
        │
        ├── survival_state
        ├── overdue_ratio
        ├── expected_next_purchase_gap
        └── survival_confidence
        │
        ▼
Stage 3: TopK detector evidence on candidate pool
        │
        ├── price_warning evidence
        ├── delivery_response evidence
        ├── terminal_dynamic evidence
        └── sales_fluctuation evidence
        │
        ▼
Stage 3.5: evidence cache / evidence timeline
        │
        ▼
Stage 4: structured evidence bundle
        │
        ▼
Future Stage 5: LLM / MCP line-card generation
```

这条链路继承之前第二阶段文档中的主数据流：清洗数据到采购事实、entity × cutoff 特征、labels、global ML、risk、business priority、TopK risk lists 和 evidence tables。

## 4. Entity 与粒度

当前实际预测对象为：

```text
entity = manufacturer_code × hospital_code × drug_group
current drug_group_source = drug_code
```

因此当前等价于：

```text
manufacturer_code × hospital_code × drug_code
```

`drug_group` 是抽象字段，未来可以升级为 `product_line_code`。`drug_category_code` 不作为当前主预测粒度，只能作为分层、上卷、辅助特征或看板分析维度。

主线必须保留：

```text
manufacturer_code
hospital_code
drug_group
drug_group_source
cutoff_month
horizon
```

如果未来是单客户单药企场景，`manufacturer_code` 可以作为租户过滤字段；如果是多药企通用系统，必须进入 entity key 或 tenant key。既有建模文档也明确提出 entity 序列可按 `manufacturer_code × hospital_code × drug_group` 聚合，并区分单药企与多药企服务模式。

## 5. Stage 1：全量风险 scorer

### 5.1 输入

```text
recurring_only entity × cutoff feature table
frequency_decay_v1 features
demand_shape_label
H3 / H6 / H12 labels during training/evaluation
```

### 5.2 输出

```text
manufacturer_code
hospital_code
drug_group
drug_group_source
cutoff_month
horizon
churn_probability_H
probability_rank_within_manufacturer
risk_band
demand_shape_label
stage1_model_version
```

### 5.3 职责

```text
全量评分；
提供概率意义；
支持概率 TopK；
支持 business priority 后处理；
给 Stage 1.5 candidate pool 提供基础输入。
```

### 5.4 非职责

```text
不输出最终业务排序；
不处理 one-shot high value；
不直接生成线索卡；
不做 detector 归因；
不替代 survival 精查。
```

## 6. Stage 1.5：业务候选池生成

### 6.1 为什么需要这一层

需求同时存在两个目标：

```text
1. 风险分需要是准确概率；
2. 最终排序需要考虑挽回价值或业务优先级。
```

因此必须拆分：

```text
churn_probability_H：概率
value_at_risk_H：潜在价值暴露
business_priority_score_H：业务排序分，不是概率
```

`business_priority_score_H` 可以参与候选池生成，但不得反向污染 `churn_probability_H` 的概率语义。

### 6.2 候选池来源

候选池不是单一概率榜，而是并集：

```text
candidate_pool =
  probability_top5pct
  ∪ business_priority_top5pct
  ∪ one_shot_high_value_attention
  ∪ demand_shape_observation_candidates
```

### 6.3 候选池字段

```text
candidate_id
manufacturer_code
hospital_code
drug_group
drug_group_source
cutoff_month
horizon
candidate_source
churn_probability_H
probability_available
value_at_risk_H
business_priority_score_H
candidate_rank
candidate_pool_top_pct
demand_shape_label
candidate_reason
```

### 6.4 one-shot high value 回归主线

one-shot high value 应该回归主线，但只能回归：

```text
candidate_pool layer
```

不能回归：

```text
probability model layer
```

one-shot 候选必须带上：

```text
candidate_source = one_shot_high_value_attention
probability_available = false
probability_interpretation = not_calibrated
recommended_route = manual_review_or_business_attention
```

这样它可以进入后续精查和证据 bundle，但不会被误解释为校准概率。

### 6.5 TopK 默认值

当前默认：

```text
Top 5%
```

保留配置项：

```text
top_candidate_pct = 0.05
```

未来可按客户调整为 Top 1%、Top 3%、Top 10%、Top N。

## 7. Stage 2：survival / interval-aware 精查

### 7.1 运行对象

只对候选池运行，不对全量 entity 运行。

```text
input = candidate_pool
```

### 7.2 职责

Stage 2 不只是再算一个概率，而是判断候选对象当前是否真的超过自身历史采购节奏。

输出风险状态：

```text
normal_interval
slightly_overdue
materially_overdue
likely_churn
low_confidence_low_frequency
not_applicable_one_shot
```

### 7.3 输出字段

```text
candidate_id
survival_method
expected_next_purchase_gap
months_since_last_purchase
overdue_ratio
survival_state
survival_confidence
survival_note
```

### 7.4 设计原则

第一版可以先做 survival-lite 或 interval-aware proxy，但接口必须保留复杂模型入口：

```text
survival_lite_overdue_ratio_v1
BG/NBD later
Pareto/NBD later
discrete-time survival later
Cox/AFT later
```

既有第二阶段文档也把 BG/NBD、Pareto/NBD 定位为后续 baseline、P_alive 特征、采购间隔特征或分路由候选算法，而不是第一版全量主线必须项。

## 8. Stage 3：TopK detector 证据层

### 8.1 运行对象

只对候选池运行。

```text
input = candidate_pool + survival_result + required facts
```

### 8.2 detector 一级分类

按照需求文档，detector family 必须按四大业务类型组织：

```text
price_warning
delivery_response
terminal_dynamic
sales_fluctuation
```

### 8.3 detector 统一输出接口

每个 detector 无论是否实现，都必须遵守统一输出：

```text
candidate_id
detector_family
detector_name
hit_flag
severity
confidence
evidence_window_start
evidence_window_end
evidence_fields
reason_code
business_interpretation
human_review_required
data_quality_note
```

### 8.4 四大类 detector catalog

#### A. price_warning

规则：

```text
low_price_purchase_warning
order_price_spread_warning
```

作用：

```text
识别低于预警价的订单；
识别同品规跨区域/跨医院价差异常；
输出价格证据、横向价格对比需求、人工核实建议。
```

v1 状态：

```text
interface only / pending reliable price data
```

#### B. delivery_response

规则：

```text
rejection_response_warning
delayed_response_warning
low_delivery_rate_warning
```

作用：

```text
识别拒绝响应、无法配送、缺货、未及时配送等；
识别确认后超时未发货；
识别配送率低于阈值；
帮助区分断供型风险与需求型流失风险。
```

v1 状态：

```text
high priority
```

#### C. terminal_dynamic

规则：

```text
terminal_loss_warning
new_terminal_detection
```

作用：

```text
识别长期未采购；
识别首次采购或 180 天以上恢复采购；
与 Stage 1 scorer 和 Stage 2 survival 精查强相关；
负责终端不丢主线。
```

v1 状态：

```text
current mainline
```

#### D. sales_fluctuation

规则：

```text
purchase_quantity_fluctuation_warning
purchase_frequency_fluctuation_warning
```

作用：

```text
识别采购量异常上升/下降；
识别采购频次异常上升/下降；
辅助判断囤货、断货、季节性需求、采购模式变化；
与终端动态联动。
```

v1 状态：

```text
medium-high priority
```

## 9. Stage 3.5：detector evidence cache

### 9.1 为什么需要 cache

同一高风险 entity 可能连续多个 cutoff 进入候选池。重复运行 detector 会产生重复成本，也会丢失连续风险演化信息。

### 9.2 cache 作用

cache 不是替代 detector，而是：

```text
复用上一期证据；
追加本期变化；
形成 evidence timeline；
记录风险是否持续或加重。
```

### 9.3 cache key

```text
manufacturer_code
hospital_code
drug_group
detector_family
detector_name
evidence_window
cutoff_month
```

### 9.4 cache 输出

```text
candidate_id
detector_name
previous_hit_flag
current_hit_flag
evidence_persistence_count
evidence_trend
new_evidence_added
cached_evidence_reference
```

## 10. Stage 4：structured evidence bundle

### 10.1 定位

Structured evidence bundle 是线索卡原材料，不是线索卡本身。

它不做自然语言夸张解释，也不生成因果断言。它只汇总结构化证据，供后续 LLM/MCP/tool 生成线索卡。

### 10.2 输入

```text
candidate_pool row
survival_result
detector_evidence_list
demand_shape_guardrail
label_confidence_note
model_limitation_note
```

### 10.3 输出字段

```text
candidate_id
entity_keys
cutoff_month
horizon
candidate_sources
churn_probability_H
business_priority_score_H
probability_interpretation
survival_state
survival_confidence
demand_shape_label
demand_shape_route
label_confidence_weight
detector_evidence_list
evidence_timeline
human_review_required
auto_dispatch_allowed
llm_allowed_facts
forbidden_claims
```

### 10.4 关键限制

```text
auto_dispatch_allowed = false
```

LLM 后续只能基于 `llm_allowed_facts` 生成表达，不允许擅自推断：

```text
竞品替代
政策落标
医院主动弃用
配送商责任
```

除非对应 detector 有明确证据。

## 11. LLM / MCP 位置

LLM 不参与：

```text
概率计算；
业务排序；
survival 判断；
detector 命中；
阈值决策。
```

LLM 只参与：

```text
把 structured evidence bundle 转化为业务可读线索卡；
生成建议动作文本；
压缩证据链；
根据用户角色调整表达。
```

未来可以封装为：

```text
MCP tool
LLM evidence summarizer
line-card generation service
```

但本阶段只设计接口，不实现 LLM。

## 12. 当前阶段不做

本阶段不做：

```text
1. 不训练新模型；
2. 不调参；
3. 不实现全部 detector；
4. 不生成正式线索卡；
5. 不自动派单；
6. 不做工单系统；
7. 不让 LLM 参与判断；
8. 不删除旧数据或 cache；
9. 不改变 probability_candidate_v1；
10. 不把 one-shot 混入概率模型训练。
```

## 14. 本阶段结论

本阶段的核心方向是：

```text
从“全量概率模型”升级为“候选池精查流水线”。
```

当前概率模型继续保留为 Stage 1 scorer。下一阶段不应直接写线索卡，也不应全量跑 detector，而应先定义：

```text
业务候选池如何生成；
one-shot 高价值如何回归主线；
business priority 如何计算但不污染概率；
survival 如何只对候选池精查；
四大类 detector 如何按统一接口输出证据；
structured evidence bundle 如何作为 LLM/线索卡原材料。
```

这就是当前阶段最合理的主思路。
