## 1. 模块定位

M8 定义 M1–M7 各模块的评估与验收协议。

M8 不是模型训练模块，也不是实验执行模块。它规定后续每个模块做完后应该如何判断“是否有效”。

## 2. Stage 1 scorer 评估

对象：

```text
probability_candidate_v1
```

指标：

```text
Brier
LogLoss
ECE
AUC
PR-AUC
Lift@K
NDCG@K
Precision@K
```

解释：

```text
Brier / LogLoss / ECE：概率质量；
AUC / PR-AUC：区分能力；
Lift / NDCG / Precision：TopK 排序辅助指标。
```

当前结论：

```text
probability_candidate_v1 = logistic_regression + frequency_decay_v1 + raw
business_usable_probability_baseline = true
```

## 3. M1 Candidate pool 评估

目标：

```text
业务排序主表是否能覆盖高风险和高价值对象。
```

指标：

```text
candidate_count
manufacturer_coverage
global_top_pct_coverage
manufacturer_min_fill_count
value_coverage_ratio
positive_rate_in_candidate_pool
manual_review_load
```

重点问题：

```text
1. Top 5% 是否过多或过少；
2. manufacturer_min_fill 是否导致低质量候选过多；
3. business_priority 是否过度偏向大金额对象；
4. H3/H6/H12 哪个适合作为 primary horizon。
```

---

## 4. M2 One-shot 评估

目标：

```text
one-shot repeat propensity 是否能识别值得关注的新终端。
```

指标：

```text
repeat_probability_H calibration
AUC / PR-AUC for second_purchase_within_H
Brier / ECE
top attention group repeat rate
high value coverage
manual review acceptance
```

需要区分三套策略：

```text
retention_risk_score
conversion_opportunity_score
balanced_attention_score
```

评估问题：

```text
1. 推荐高不复购风险对象是否更有挽回价值；
2. 推荐高复购概率对象是否更适合促成转化；
3. balanced score 是否更适合作为默认；
4. one-shot 解释是否能让业务理解。
```

需求侧未确认前，M8 不强行确定最终策略。

## 5. M3 Survival-lite 评估

目标：

```text
survival-lite 是否能更准确识别候选对象是否超出自身采购节奏。
```

指标：

```text
survival_state 分布
survival_state 与后续 die_H 的关系
overdue_ratio 分箱后的 observed die rate
history_sufficiency 覆盖率
survival_confidence 与准确性的关系
```

验收重点：

```text
1. materially_overdue / likely_churn_interval 的实际 die rate 是否高于 normal_interval；
2. insufficient_history 是否确实应低置信；
3. intermittent / lumpy 是否不适合短 H 强预警；
4. group prior fallback 是否产生过多误判。
```

## 6. M4 Detector 评估

目标：

```text
detector 是否能提供业务可解释证据，而不是重新制造一个黑盒分数。
```

指标：

```text
hit_rate
severity 分布
confidence 分布
人工复核认可率
detector evidence 与后续 die_H 的关联
data_quality_not_evaluable 占比
```

按 family 分别评估：

```text
price_warning
delivery_response
terminal_dynamic
sales_fluctuation
```

v1 重点评估：

```text
terminal_loss_warning
purchase_frequency_fluctuation_warning
purchase_quantity_fluctuation_warning
new_terminal_detection
```

配送类和价格类若仅保留接口，则不要求效果评估，只要求 data_quality_note 清楚。

## 7. M5 状态决策评估

目标：

```text
final_candidate_status 是否符合业务处理优先级。
```

指标：

```text
priority_review 数量
manual_review 数量
observation_only 数量
low_confidence_watch 数量
P0/P1/P2/P3 分布
各状态后续 die_H positive rate

验收问题：

2. low_confidence_watch 是否避免了低质量强预警；
3. demand-shape guardrail 是否降低误报；
4. auto_dispatch_allowed 是否保持 false。
```

## 9. M7 Structured evidence bundle 评估

目标：

```text
structured evidence bundle 是否足以支持后续线索卡生成。
```

指标：

```text
bundle completeness
allowed_claims completeness
forbidden_claims coverage
detector_evidence_list completeness
missing critical fields rate
manual readability check
```

验收问题：

```text
1. 是否包含 probability / business priority / survival / detector / guardrail；
2. 是否明确哪些可以说、哪些不能说；
3. 是否能生成线索卡但不做无依据因果断言；
4. 是否区分 recurring 与 one-shot。
```

## 10. 真实业务回测要求

最终进入业务试点前，需要真实客户样本回测。

最小回测集：

```text
5–10 个真实流失终端
一组同群正常终端
完整历史订单
客户确认的流失时间或流失口径
```

评估：

```text
提前多少天进入 priority_review；
是否有 survival 超期证据；
是否有 detector 证据；
是否被 demand-shape guardrail 正确处理；
线索卡材料是否能被业务理解。
```
