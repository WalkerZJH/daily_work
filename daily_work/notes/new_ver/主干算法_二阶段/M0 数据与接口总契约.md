## 0.1 模块定位

M0 是后续所有模块的共同契约层。它不做模型、不做排序、不做 detector，只规定：

```text
1. 什么是一个 entity；
2. 什么是一个 cutoff；
3. 哪些表是主表，哪些表是旁路关注表；
4. 每个字段的语义；
5. 哪些字段可解释为概率，哪些不能；
6. 哪些对象进入后续 survival / detector；
7. 哪些对象只进入首页关注或观察区。
```

M0 的作用是防止后续模块混乱，尤其是避免：

```text
1. 把 business_priority_score 当概率；
2. 把 one-shot attention 当 recurring churn probability；
3. 把 demand-shape observation 当正式高风险；
4. 把 drug_group 误解成 drug_category_code。
```

当前配置中 entity grain 是 `manufacturer_code / hospital_code / drug_group`，并且 `primary_drug_group_source` 是 `drug_code`。因此当前实际预测粒度是 `manufacturer_code × hospital_code × drug_code`，不是药品大类预测。

## 0.2 全局主键

所有主线表必须保留以下字段：

```text
manufacturer_code
hospital_code
drug_group
drug_group_source
cutoff_month
```

其中：

```text
drug_group_source = drug_code
```

当前实际 entity 为：

```text
manufacturer_code × hospital_code × drug_code
```

如果未来客户提供产品线映射，可升级为：

```text
drug_group_source = product_line_code
```

但当前不得写成：

```text
drug_group_source = drug_category_code
```

`drug_category_code` 只能作为辅助维度、上卷维度、分层统计维度或看板维度。

## 0.3 Horizon 契约

当前仍保留三个预测窗口：

```text
H3
H6
H12
```

每个 horizon 都有自己的：

```text
churn_probability_H
value_at_risk_H
business_priority_score_H
risk_band_H
label_confidence_H
```

注意：**M1 不应该强行把 H3/H6/H12 压成一个唯一总分**。否则会丢失业务语义。

正确做法是：

```text
1. 先分别计算 business_priority_score_H3 / H6 / H12；
2. 再由展示或客户配置决定 primary_priority_horizon；
3. 默认可以先用 H6 作为主页主视图；
4. H3 作为短期急迫风险；
5. H12 作为长期流失/低频关系观察风险。
```

这样可以解决你提到的“评分窗口固定”的问题：底层不固定，前端或报表可以固定一个默认视角。

## 0.4 对象类型契约

后续至少存在三类对象，不允许混成一个表统一评分。

### A. recurring business priority candidate

正常主线对象。来自 recurring entity。具备：

```text
churn_probability_H
value_at_risk_H
business_priority_score_H
```

可进入：

```text
survival 精查
detector 精查
structured evidence bundle
```

### B. one-shot high value attention

单次采购高价值关注对象。它不是 recurring churn probability 模型对象。

不允许声称：

```text
该对象的 churn_probability_H = xx
```

可以输出：

```text
one_shot_attention_score
one_shot_value_score
one_shot_similarity_risk_score
```

但必须标注：

```text
not_calibrated_churn_probability
```

### C. demand-shape observation

特殊需求形态观察对象。主要包括：

```text
intermittent
lumpy
H3 不适合强预警的低频对象
```

它不是主业务排序高位对象，也不是 one-shot。它用于首页观察区或人工复核区。

## 0.5 三张核心输出表

M0 规定下一阶段至少输出三张逻辑表。

### 表 1：recurring_business_priority_candidates

主表。只放正常 recurring 的业务排序高位候选。

```text
reports 或后续 artifact:
recurring_business_priority_candidates
```

它的排序依据是：

```text
business_priority_score_H = churn_probability_H × value_at_risk_H
```

它是后续 survival 和 detector 的主输入。

### 表 2：one_shot_attention_candidates

旁路关注表。只放 one-shot 高价值对象。

```text
one_shot_attention_candidates
```

它不参与 recurring business priority 排序，不进入主概率模型。

### 表 3：demand_shape_observation_candidates

观察表。只放因 demand-shape guardrail 需要关注但不应强预警的对象。

```text
demand_shape_observation_candidates
```

它不参与主业务优先级排序，不与主候选池取并集。

---

## 0.6 展示层去重规则

虽然三张表逻辑分离，但首页展示时可能出现极少数重复对象。展示层去重规则为：

```text
business_priority > demand_shape_observation > one_shot_attention
```

也就是说：

```text
如果某 entity 已进入 business_priority 主表，则不再在 observation / one-shot 区重复展示。
如果某 entity 同时进入 observation 和 one-shot，则保留 observation。
```

但去重只影响展示，不改变底层审计记录。

---

## 0.7 概率语义约束

以下字段可以解释为概率：

```text
churn_probability_H3
churn_probability_H6
churn_probability_H12
```

以下字段不能解释为概率：

```text
business_priority_score_H
value_at_risk_H
one_shot_attention_score
one_shot_similarity_risk_score
demand_shape_observation_score
detector_severity
```

所有文档和 Notebook 必须统一这个语义。

---

## 0.8 后续模块输入关系

M0 规定：

```text
M1 recurring_business_priority_candidates
    → M3 survival
    → M4 detector
    → M7 structured evidence bundle

M2 one_shot_attention_candidates
    → one-shot 专项解释
    → 可进入 structured evidence bundle，但不进入 recurring survival

M1 demand_shape_observation_candidates
    → observation review
    → 可进入 structured evidence bundle，但不作为高风险主候选
```

也就是说，后续 survival / detector 的主处理对象是 **业务排序主表**，不是三表并集。
