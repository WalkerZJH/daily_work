## 总体模块拆分

```text
M0 数据与接口总契约
M1 Stage 1.5 候选池生成与业务排序
M2 One-shot high value 特殊方案
M3 Survival / interval-aware 精查模块
M4 Detector catalog 与各 detector 算法方案
M5 证据融合与风险状态决策
M6 Detector evidence cache / evidence timeline
M7 Structured evidence bundle / LLM 输入材料
M8 回测、评估与验收协议
```

其中，**M1、M2、M3 是下一阶段最核心**。Detector 不能先做，因为 detector 的运行对象来自候选池；Survival 也不能独立设计，因为它只对候选池 TopK 精查。

---

# M0：数据与接口总契约

这个模块先做，但可以很短。它定义整条流水线的统一 key、输入输出表和命名规范。

核心要确定：

```text
entity_key = manufacturer_code × hospital_code × drug_group
drug_group_source = drug_code
cutoff_month
horizon = H3 / H6 / H12
candidate_source
```

它还要明确不同层输出的含义：

```text
churn_probability_H：概率模型输出
business_priority_score_H：业务排序分，不是概率
one_shot_attention_score：one-shot 关注分，不是 recurring churn probability
survival_state：精查状态，不一定是概率
detector_evidence：证据，不是最终分数
structured_evidence_bundle：线索卡原材料
```

这一层的目标是避免后续模块语义混乱。

---

# M1：候选池生成与业务排序模块

这是下一步最优先设计的模块。

你现在已经明确：高位候选不应只是“高流失概率”，还应包括：

```text
高流失概率 × 高价值
one-shot high value
特殊 demand-shape 观察对象
```

因此 M1 要设计：

```text
candidate_pool =
  probability_top5pct
  ∪ business_priority_top5pct
  ∪ one_shot_high_value_attention
  ∪ demand_shape_observation_candidates
```

其中 recurring 部分可以有两个排序：

```text
rank_by_probability
rank_by_business_priority
```

业务排序分可以先定义为：

```text
business_priority_score_H = churn_probability_H × value_at_risk_H
```

但要明确：

```text
business_priority_score_H 不是概率；
它只决定是否进入候选池和后续精查优先级；
不得用于评价概率模型校准。
```

M1 的关键问题是：

1. `value_at_risk_H` 用什么计算？
    
2. Top 5% 是按 manufacturer 内排序，还是全局排序？
    
3. 多个来源重复命中时如何合并？
    
4. one-shot high value 如何并入但不污染概率模型？
    
5. 候选池规模是否按 `top_pct`、`top_n`、还是二者结合？
    

这个模块做完后，后续 survival 和 detector 才知道自己处理哪些对象。

# M2：One-shot high value 特殊方案

你对 one-shot 的判断是对的：**one-shot 几乎不能使用 recurring churn model，也很难直接得到同语义概率**。但它不能被丢在主线之外，因为它有业务价值。

我建议给 one-shot 单独设计一个模块，名称不要叫 churn probability，可以叫：

```text
one_shot_attention_score
```

或者更严格：

```text
one_shot_repeat_failure_risk
```

这里有两种方案。

## 方案 A：纯规则 attention score

适合第一版，稳定、可解释。

输入：

```text
hospital_level_code
ownership_type_code
province_code / city_code
drug_code / drug_category_code
manufacturer_code
first_purchase_amount
first_purchase_quantity
delivery_rate / arrival_rate
order_status
```

输出：

```text
one_shot_attention_score
one_shot_value_score
one_shot_candidate_flag
```

缺点是它不是概率。

## 方案 B：冷启动 repeat propensity 模型

这个更像算法方案。不是预测“流失”，而是预测：

```text
P(second_purchase_within_H | first_purchase_context)
```

然后：

```text
one_shot_non_repeat_risk_H = 1 - P(second_purchase_within_H)
```

训练数据来自历史 one-shot / first-purchase 关系。每个首次采购 entity，在首次采购后的 H 窗口内是否发生第二次采购。

这个方案更合理，因为 one-shot 不是 recurring churn，而是：

```text
首次采购后是否能转化为持续采购关系
```

它的特征可以包括：

```text
医院等级
地区
所有制
药品类别
具体药品
生产企业
首次采购金额/数量
配送是否完整
是否退回/作废/失败
首次采购月份
同地区/同等级医院历史复购率
同药品历史复购率
```

输出：

```text
one_shot_repeat_probability_H
one_shot_non_repeat_risk_H
one_shot_attention_score_H =
    one_shot_non_repeat_risk_H × first_or_expected_value
```

但必须标注：

```text
这不是 recurring churn_probability_H。
```

我的建议：**M2 先设计双路径**。v1 用规则 attention score；v2 保留 repeat propensity 模型接口。

# M3：Survival / interval-aware 精查模块

Survival 模块应该只对 M1 生成的候选池运行，不跑全量。

它的核心问题不是“重新排序所有 entity”，而是回答：

```text
这个候选对象是否真的超过了自己的正常采购节奏？
```

M3 可以分三层设计。

## M3-v1：survival-lite / interval-aware proxy

第一版建议先做这个，而不是直接上复杂 survival 模型。

输入：

```text
months_since_last_purchase
median_purchase_interval
mean_purchase_interval
ADI
CV²
purchase_count_asof_cutoff
active_month_count_asof_cutoff
demand_shape_label
order_count_last_3m/6m/12m
```

核心输出：

```text
expected_next_purchase_gap
overdue_ratio
interval_percentile
survival_state
survival_confidence
```

例如：

```text
overdue_ratio =
months_since_last_purchase / expected_purchase_interval
```

状态：

```text
normal_interval
slightly_overdue
materially_overdue
likely_churn
low_confidence_low_frequency
not_applicable_one_shot
```

这个非常适合和 demand-shape routing 结合。

## M3-v2：BTYD / BG-NBD / Pareto-NBD

适合非契约购买行为，但在医院采购场景要小心，因为政策、集采、配送因素会破坏独立随机购买假设。

它可以作为：

```text
P(alive) 辅助精查
expected future purchase count
expected next purchase time
```

但不建议一开始作为主实现。

## M3-v3：离散时间 survival / hazard model

如果后续数据充足，最贴合当前 `entity × cutoff` 数据结构的是：

```text
discrete-time survival model
```

它可以把每个 cutoff 当作时间片，预测 hazard：

```text
P(event occurs in next interval | survived so far)
```

但这比 survival-lite 更复杂，需要更严谨的样本构造和 censoring 处理。

我的建议：M3 详细设计时按三档写：

```text
v1: survival-lite
v2: BG/NBD / Pareto-NBD proxy
v3: discrete-time survival
```

先实现 v1，保留 v2/v3 接口。

# M4：Detector catalog 与算法方案

这一模块要完整覆盖四大业务类别，但不要求全部实现。

四大类应按需求侧口径组织：

```text
1. price_warning
2. delivery_response
3. terminal_dynamic
4. sales_fluctuation
```

每个 detector 都要统一输出：

```text
detector_family
detector_name
hit_flag
severity
confidence
evidence_window
reason_code
evidence_fields
business_interpretation
human_review_required
```

## 价格预警类

包括：

```text
低价采购预警
订单价差异常
```

第一版建议只定义接口，不实现或弱实现。原因是采购价可能被独立脱敏，真实价差判断不可靠。

更适合的方案是：

```text
rule-based + 业务配置阈值
```

未来如果有真实可比价数据，再做规则增强。

## 配送响应类

包括：

```text
拒绝响应预警
响应不及时预警
配送率低预警
```

第一版优先级高。

适合：

```text
规则 + 简单统计
```

例如：

```text
delivery_rate < 0.8
confirmed but no delivery after N days
delivery_state_code indicates failed/refused
```

后续可扩展为配送企业级系统性异常检测。

## 终端动态类

包括：

```text
终端丢失预警
新进终端识别
```

这里与 Stage 1 scorer、M2 one-shot、M3 survival 重叠最大。

终端丢失应主要由：

```text
Stage 1 probability + Stage 2 survival state
```

给结论，detector 只补充证据。

新进终端则由 one-shot 模块处理。

## 销量波动类

包括：

```text
采购量异常波动
采购频次异常波动
```

适合：

```text
规则 + robust statistics
```

例如：

```text
recent quantity vs historical median
recent frequency vs historical frequency
MAD / IQR / percentile
```

不建议一开始用复杂机器学习，因为 detector 主要服务证据链。规则更可解释。
