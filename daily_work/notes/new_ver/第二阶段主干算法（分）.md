# 下一阶段主干算法设计思路：医院-药品采购序列建模

## 一、阶段目标

基于已冻结的 `BS_Agent_DingDan model_base`，构建面向“终端不丢 / 用户存活”的主干算法方案。

本阶段不直接训练正式模型，重点完成：

1. 分析单元定义；
2. 采购序列构建；
3. 时间分桶粒度调研；
4. 流失 / 存活标签设计；
5. 特征工程方案设计；
6. 候选算法路线比较；
7. 评价指标设计；
8. 后续 feature builder 输入输出规范。

## 二、核心分析单元

初步分析单元：

```text
hospital_code × drug_group
```

其中 `drug_group` 待确认，可选口径：

```text
1. drug_code：医院 × 单药品
2. drug_category_code：医院 × 药品类别
3. product_line_code：医院 × 产品线，需额外映射
4. manufacturer_code × drug_code：医院 × 本企业药品
```

待确认问题：

```text
- 当前 drug_category_code 是否能代表业务产品线？
- 是否需要客户提供 SKU → 产品线映射表？
- 风险分应建在 医院 × 单药品，还是 医院 × 产品线？
- 证据链是否需要下钻到具体 drug_code / 订单明细？
```

## 三、采购行为口径

第一版只考虑采购行为。

只要某个分析单元在观察窗口后继续发生采购行为，即认为该医院-药品关系仍然存活。

暂不将以下情况判为死亡：

```text
- 拒绝配送
- 无法配送
- 配送失败
- 未到货
- 退货
```

这些状态可作为后续风险解释或履约类特征，但不作为第一版 alive/die 标签的核心依据。

## 四、采购序列构建

从 `model_base` 按以下字段构建序列：

```text
entity_key = hospital_code × drug_group
time_key = purchase_time
event = 是否发生采购
value_quantity = raw_sensitive_purchase_quantity
value_amount = raw_sensitive_purchase_amount
order_status = order_phase_code / delivery_state_code
```

每个 entity 生成按时间排序的采购事件序列：

```text
[(purchase_time_1, quantity_1, amount_1),
 (purchase_time_2, quantity_2, amount_2),
 ...]
```

再根据调研结果选择时间分桶：

```text
候选 grain：
- week
- month
- quarter
- 自适应分桶：按历史采购间隔 / ADI 决定
```

月度分桶是当前工程上最自然的起点，但其合理性需要通过采购间隔分布验证，不能直接固定。

## 五、需求形态分类

在正式建模前，必须先判断每个 entity 的采购序列形态。

候选指标：

```text
ADI = 平均采购间隔
CV² = 采购量变异系数平方
purchase_interval_median
purchase_interval_iqr
active_month_ratio
seasonality_strength
```

初始分类：

```text
smooth：稳定采购
erratic：高波动采购
intermittent：间断采购
lumpy：块状 / 突发采购
seasonal：季节性采购
burst：爆发性采购
cold_start：历史不足
```

不同序列形态后续应使用不同特征、不同阈值、甚至不同算法路线。

第一版可以先不实现复杂路由，但设计文档中必须保留路由思想。

## 六、标签设计

候选标签：

```text
alive_in_H = 未来 H 时间窗口内是否发生采购
die_in_H = 未来 H 时间窗口内没有发生采购
time_to_next_purchase = 距离下一次采购的天数
overdue = 当前距离上次采购时间是否超过历史期望间隔
```

预测窗口 H 不应只固定为 3 个月。

候选 H：

```text
H = 3 个月
H = 6 个月
H = 12 个月
H = 自适应窗口：历史中位采购间隔 × k
```

待调研问题：

```text
- 对稳定慢病药，3 个月窗口是否足够？
- 对季节性药，6-12 个月窗口是否更合理？
- 对突发采购药，是否不适合用固定窗口定义流失？
- 是否应根据需求形态选择不同 H？
```

## 七、第一版特征工程方向

### 1. 采购节奏特征

```text
days_since_last_purchase
median_purchase_interval
mean_purchase_interval
std_purchase_interval
purchase_interval_cv
orders_last_1_bucket
orders_last_3_bucket
orders_last_6_bucket
active_bucket_count_last_6
```

### 2. 采购频次特征

```text
order_count_last_3
order_count_last_6
order_count_last_12
frequency_recent_vs_base
active_ratio_recent_vs_base
```

### 3. 采购数量 / 金额特征

```text
purchase_quantity_sum_last_3
purchase_quantity_sum_last_6
purchase_amount_sum_last_3
purchase_amount_sum_last_6
quantity_recent_vs_base
amount_recent_vs_base

```

注意：

```text
允许比较数量趋势；
允许比较金额趋势；
禁止 金额 / 数量 推断真实单价。
```

### 4. 关系强度特征

```text
hospital_drug_total_orders
hospital_drug_active_months
hospital_total_orders
drug_total_hospitals
hospital_drug_share_in_hospital
```

### 5. 静态特征

```text
province_code
city_code
county_code
hospital_level_code
ownership_type_code
drug_category_code
manufacturer_code
```

### 6. 状态辅助特征

```text
submitted_count

received_count
failed_count

```

第一版标签只看采购行为，因此状态字段不决定 alive/die，但可用于解释和辅助建模。若后续预测目标改为履约质量，则这些字段需要重新审查泄漏风险。

## 八、候选算法路线

第一优先级：传统机器学习 / 表格模型。

候选：

```text
LightGBM
CatBoost
XGBoost
Logistic Regression baseline
Random Forest baseline
```

备选统计主干：

```text
BG/NBD
Pareto/NBD
```

暂不优先：

```text
深度学习
纯规则模型
复杂 agent 推理
```

建议策略：

```text
1. 用 LightGBM / CatBoost 做第一版 ML 主线；
2. 用 BG/NBD 或 Pareto/NBD 作为统计基准或特征来源；
3. 后续根据序列形态引入路由算法；
4. 对不同需求形态分别比较模型效果。
```

## 九、评价指标设计

第一版不能只看 accuracy / AUC。

排序型指标应优先，因为业务关心的是“Top K 高风险对象中哪些真的流失”。

候选指标：

```text
Precision@K
Recall@K
NDCG@K
Lift@K
Top-K captured churn value
Brier score
Calibration curve
Time-to-event aware NDCG
```

需要特别处理预测窗口：

```text
- 如果 H 太短，季节性药品可能被误判为流失；
- 如果 H 太长，预警提前性变差；
- 可以同时评估 H=3/6/12 个月；
- 或根据需求形态使用自适应 H。
```
