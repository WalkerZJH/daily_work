# 用户存活 / 终端不丢指标设计参考

## 1. 目标

本任务不是普通分类问题，而是风险排序问题。

业务目标是：在所有 `医院 × 药品/产品线` 单元中，找出最值得优先关注的高流失风险对象。

因此，评价指标应优先回答：

```text
预测排名靠前的 Top K 对象中，有多少真的在后续观察窗口内停止采购？
真正停止采购的对象，是否被排在足够靠前的位置？
模型是否能比随机排序、简单规则排序更早发现风险？
```

## 2. 样本单元

建议第一版样本单元为：

```text
entity = hospital_code × drug_group
cutoff_time = 某个历史评估时点
```

其中 `drug_group` 可为：

```text
drug_code
drug_category_code
product_line_code
manufacturer_code × drug_code
```

最终口径需结合业务确认。

## 3. 标签口径

第一版只考虑采购行为。

```text
alive_in_H = cutoff_time 后 H 窗口内存在采购事件
die_in_H = cutoff_time 后 H 窗口内不存在采购事件
```

采购事件定义：

```text
purchase_time 非空
order_detail_id 有效
```

暂不因为拒绝配送、无法配送、退货、未到货等状态将该订单排除。只要发生采购，就认为该 entity 在该窗口中仍然活跃。

## 4. 预测窗口 H

固定 H 存在风险：

```text
H 太短：季节性药品可能被误判为流失；
H 太长：预警提前性下降；
不同药品采购周期差异很大，统一窗口可能不公平。
```

建议同时评估多个窗口：

```text
H = 3 个月
H = 6 个月
H = 12 个月
```

后续可引入自适应窗口：

```text
H_entity = max(业务最小窗口, 历史中位采购间隔 × k)
```

或按需求形态设置窗口：

```text
smooth：较短 H
erratic：中等 H
intermittent：较长 H
lumpy：低置信或特殊路由
seasonal：至少覆盖一个季节周期
```

## 5. 基础排序指标

### 5.1 Precision@K

```text
Precision@K = Top K 中真实流失对象数 / K
```

含义：

```text
如果业务员只能处理 K 条线索，Top K 中有多少是真风险？
```

适合衡量线索质量。

建议评估：

```text
Precision@10
Precision@20
Precision@50
Precision@100
```

### 5.2 Recall@K

```text
Recall@K = Top K 中真实流失对象数 / 全部真实流失对象数
```

含义：

```text
真实流失对象中，有多少被模型排进了前 K？
```

适合衡量覆盖能力。

### 5.3 Lift@K

```text
Lift@K = Precision@K / 全局流失率
```

含义：

```text
模型 Top K 相比随机选择提升了多少倍？
```

例如全局流失率 5%，Top 100 Precision 为 25%，则：

```text
Lift@100 = 25% / 5% = 5
```

说明 Top 100 的风险密度是随机的 5 倍。

## 6. NDCG@K

NDCG 适合评价“真风险是否排得足够靠前”。

基础形式：

```text
DCG@K = Σ gain_i / log2(i + 1)
NDCG@K = DCG@K / IDCG@K
```

其中：

```text
i = 排名位置
gain_i = 第 i 个对象的收益
```

第一版可设：

```text
gain_i = 1，如果对象在 H 窗口内流失
gain_i = 0，如果对象在 H 窗口内未流失
```

这样，真实流失对象排得越靠前，NDCG 越高。

## 7. 加权 NDCG

实际业务中，不同终端价值不同。可以引入价值权重：

```text
gain_i = churn_label_i × terminal_value_weight_i
```

terminal_value_weight 可来自：

```text
历史采购金额
历史采购数量
医院等级
药品价值档
客户自定义重点终端
```

这样模型不仅奖励“找到流失”，还奖励“优先找到高价值流失”。

## 8. 时间感知 NDCG

固定窗口会误伤季节性和低频采购对象。可以将 gain 设计为时间感知版本。

候选：

```text
gain_i = churn_label_i × time_weight_i
```

其中：

```text
time_weight_i = 1 / log(2 + days_to_churn_i)
```

含义：

```text
越早真实流失，且模型越早排到前面，奖励越高；
很久以后才停购的对象，奖励较低；
未来很快复购的对象，如果排很靠前，应被惩罚。
```

第一版可先不实现复杂时间感知 NDCG，但应在指标设计中保留该方向。

## 9. 多窗口指标

为了避免单一 H 导致误判，建议所有排序指标都按多个 H 输出：

```text
Precision@K_H3
Precision@K_H6
Precision@K_H12

Recall@K_H3
Recall@K_H6
Recall@K_H12

NDCG@K_H3
NDCG@K_H6
NDCG@K_H12

Lift@K_H3
Lift@K_H6
Lift@K_H12
```

如果模型只在 H=3 表现好，但在 H=6/H=12 明显变差，说明它可能只是识别短期断购，不适合季节性或长周期药品。

## 10. 提前量指标

终端不丢的核心价值是“提前预警”。

建议衡量：

```text
lead_time = 真实停购日 - 首次进入 Top K 或首次超过风险阈值的日期
```

输出：

```text
提前 ≥30 天命中率
提前 ≥60 天命中率
提前 ≥90 天命中率
lead_time 中位数
lead_time P25 / P75
```

## 11. 校准指标

如果模型输出的是概率或风险分，还需要检查分数是否名副其实。

建议指标：

```text
Brier Score
Calibration Curve
ECE
分箱实际流失率
```

即使业务最终看 Top K，校准也重要，因为风险等级、红橙黄阈值依赖概率解释。

## 12. 业务推荐的第一版指标组合

第一版建议固定以下指标：

```text
主指标：
- Precision@K
- NDCG@K
- Lift@K

辅助指标：
- Recall@K
- 多窗口 H=3/6/12 对比
- 提前 ≥30/60/90 天命中率
- Brier Score / Calibration Curve

分组指标：
- 按 drug_group
- 按 hospital_level_code
- 按需求形态 smooth / erratic / intermittent / lumpy / seasonal
```

## 13. 暂不建议作为主指标

```text
Accuracy
普通 F1
随机 K 折 AUC
只看单一 H 的分类指标
```

原因：

```text
流失样本通常不平衡；
业务关注 Top K 而不是全量平均准确率；
随机 K 折会造成时间穿越；
单一 H 会误伤长周期或季节性采购。
```

## 14. 待确认问题

```text
1. K 取多少？对应 VP / 业务员每天能处理多少线索？
2. H 是否固定，还是按需求形态自适应？
3. 真实流失标签是否只看采购行为？
4. 高价值终端如何定义？
5. 是否需要客户人工确认真实流失样本？
6. 评价对象是 医院×药品，还是 医院×产品线？
```
