# 用户存活 / 终端不丢主干算法数据流设计

## 1. 文档目标

本文描述从 `BS_Agent_DingDan model_base` 到用户存活风险排序模型的主干数据流。
注意，数据取得可以使用algo_main/src/alg/cleaning/bs_agent_dingdan_pipeline.py中的run_bs_agent_dingdan_cleaning_pipeline，它可以有缓存的提取相关数据，目前暂时已经缓存5W行数据。
当前阶段目标不是直接处理全量数据训练复杂模型，而是先设计稳定、可分层、可回测的数据流，使后续可以逐步扩展：

```text
model_base
→ 采购事实表
→ 医院×药品/产品线×时间切片
→ cutoff 训练样本
→ 多窗口标签
→ 全局模型训练
→ 风险分与业务优先级排序
→ 按药企分组输出 TopK 风险清单
```

## 2. 总体理解

本任务的建模对象不是单条订单，也不是单个医院整体，而是：

```text
entity = hospital_code × drug_group
```

其中 `drug_group` 可以是：

```text
drug_code
drug_category_code
product_line_code
manufacturer_code × drug_code
manufacturer_code × product_line_code
```

第一版如果没有明确产品线映射，可以先使用：

```text
entity = hospital_code × drug_code
```

后续再升级为：

```text
entity = hospital_code × product_line_code
```

## 3. 任务形态转变

初始直觉可能是：

```text
预测某个 entity 的下一次采购时间
```

但当前更适合转为：

```text
在某个 cutoff 时点，对所有 entity 进行流失风险排序
```

因此任务从“纯生存模型 / 时间到事件预测”转向：

```text
推荐 / 排序问题：
推荐最需要业务人员跟进的高流失风险 entity
```

即：

```text
输入：截至 cutoff 的历史采购行为
输出：未来 H 窗口内的流失风险，以及业务优先级排序
```

生存模型、BG/NBD、Pareto/NBD 仍可作为：

```text
1. baseline；
2. P_alive 特征；
3. 采购间隔特征；
4. 后续分路由模型的候选算法。
```

但第一版主线建议采用全局表格模型。

### 4.1 原始清洗层

```text
data/03_cleaned/bs_agent_dingdan_model_base.parquet
```

注意：

```text

```

### 4.2 采购事件层

从订单行中抽取采购事件：

```text
purchase_event = purchase_time 非空 且 order_detail_id 有效
```

第一版只考虑采购行为：

```text
只要发生采购，就认为该 entity 仍然活跃。
```

暂不因为以下状态否定采购事件：

```text
拒绝配送
无法配送
配送失败
退货
未到货
```

这些状态作为历史风险特征或履约解释特征，而不是 alive/die 标签本身。

输出建议：

```text
data/04_facts/fact_purchase_event.parquet
```

字段建议：

```text
row_uid
order_detail_id
manufacturer_code
hospital_code
drug_code
drug_group
purchase_time
purchase_month
raw_sensitive_purchase_quantity
raw_sensitive_purchase_amount
order_phase_code
delivery_state_code
order_failure_flag
order_terminal_flag
```

### 4.3 entity 采购序列层

按 entity 聚合订单，形成采购序列：

```text
entity_key = manufacturer_code × hospital_code × drug_group
```

注意是否包含 `manufacturer_code` 取决于服务模式：

```text
如果服务单个药企：manufacturer_code 可作为租户过滤字段；
如果服务多药企：manufacturer_code 必须进入 entity_key 或 tenant_key。
```

每个 entity 的序列：

```text
[
  (purchase_time_1, quantity_1, amount_1, status_1),
  (purchase_time_2, quantity_2, amount_2, status_2),
  ...
]
```

输出建议：

```text
data/04_facts/entity_purchase_sequence.parquet
```

字段建议：

```text
tenant_key
entity_key
manufacturer_code
hospital_code
drug_group
first_purchase_time
last_purchase_time
purchase_count_total
active_month_count_total
purchase_time_list 或 sequence_index
```

如果不存 list，可存事件明细表，通过 `entity_key + purchase_time` 排序复原。

### 4.4 月度事实层

第一版建议采用月度分桶作为工程起点：

```text
bucket = purchase_month
```

但月度不是最终理论结论，只是 v1 的统一时间网格。后续需要通过采购间隔分布调研决定是否改为周、季度或自适应窗口。

输出建议：

```text
data/04_facts/fact_hospital_drug_month.parquet
```

主键：

```text
manufacturer_code
hospital_code
drug_group
purchase_month
```

字段建议：

```text
order_count
purchase_quantity_sum
purchase_amount_sum
delivery_quantity_sum
delivery_amount_sum
arrival_quantity_sum
arrival_amount_sum

submitted_count
confirmed_count
received_count
failed_count
terminal_count
rejected_or_failed_count

has_purchase
has_failed_order
has_received_order

```

没有采购的月份是否补 0，需要根据训练集构造阶段决定。通常为了构造时间序列特征，需要补齐 entity 从首次采购到 cutoff 的连续月份。

## 6. cutoff 样本层

模型训练样本不应是一条原始订单，也不应是一整条完整序列，而应是：

```text
entity × cutoff_month
```

示例：

```text
entity = A医院 × B药品
cutoff_month = 2025-06
```

样本含义：

```text
截至 2025-06 这个时点，A医院-B药品 的历史采购状态如何？
未来 H 窗口内是否还会采购？
```

输出建议：

```text
data/05_features/alive_prediction_cutoff_samples.parquet
```

主键：

```text
manufacturer_code
hospital_code
drug_group
cutoff_month
```

特征只允许使用：

```text
cutoff_month 及以前的数据
```

禁止使用：

```text
cutoff_month 之后的订单、状态、数量、金额、配送、到货信息
```

### 7.1 采购节奏特征

### 7.2 频次特征

### 7.3 数量 / 金额特征


注意：

```text

禁止使用 金额 / 数量 推断真实单价。
```

### 7.4 状态历史特征

状态字段不能直接作为未来标签泄露，但 cutoff 以前的状态统计可以作为历史信号。

```text
failed_count_last_3m
failed_count_last_6m
failed_rate_last_6m
days_since_last_failed_order

received_rate_last_6m

last_delivery_state_code_asof_cutoff
last_order_failure_flag_asof_cutoff
```

规则：

```text
只使用 cutoff 以前的状态；
不使用预测窗口内状态；
如果预测目标改为“未来是否失败/到货/终止”，则相关状态字段需要重新审查泄漏。
```

### 7.5 静态实体特征

```text
hospital_code
drug_group
province_code
```

```text

```

### 7.6 需求形态特征


## 8. 标签层

第一版多固定窗口标签：

```text
H = 3个月
H = 6个月
H = 12个月
```

标签定义：

```text
alive_H = cutoff 后 H 窗口内是否存在采购事件
die_H = 1 - alive_H
```

对应字段：

```text
label_alive_H3
label_die_H3

label_alive_H6
label_die_H6

label_alive_H12
label_die_H12
```

注意：

```text
标签只看采购行为；
只要未来窗口内发生采购，即认为 alive；
不因拒绝配送、无法配送、退货、未到货而判定死亡。
```

后续可引入：

```text
dynamic_H
adaptive_die_label
time_to_next_purchase
days_until_next_purchase
```

但 v1 不作为主标签。

## 9. 模型训练层

第一版采用全局模型，而不是每个 entity 独立训练。

训练样本：

```text
manufacturer_code × hospital_code × drug_group × cutoff_month
```

候选模型：

```text
LightGBM
CatBoost
XGBoost
Logistic Regression baseline
```

暂不优先：

```text
深度学习
联邦学习
每个 entity 独立模型
复杂路由模型
```

训练目标可以有两种方式：

### 9.1 多模型

分别训练：

```text
model_H3
model_H6
model_H12
```

输出：

```text
risk_H3
risk_H6
risk_H12
```

### 9.2 多目标模型

一个模型同时输出多个窗口风险：

```text
risk_H3
risk_H6
risk_H12
```

v1 建议先使用多模型，便于评估和调试。

## 10. 风险分与业务排序层

模型输出的纯风险分：

```text
risk_score_H = P(die_H = 1 | features_asof_cutoff)
```

但业务排序不应只看风险概率。建议再计算：

```text
business_priority_score_H
= risk_score_H
  × expected_value_at_risk_H
  × urgency_weight_H
  × confidence_weight
```

其中：

```text
expected_value_at_risk_H = 基于历史采购金额/数量估计的未来可能损失
urgency_weight_H = 风险越近权重越高
confidence_weight = 样本越充分、模型越可靠权重越高
```

第一版可以简化为：

```text
business_priority_score_H = risk_score_H × expected_value_at_risk_H
```

然后逐步加入紧迫性和置信度。

## 11. 多药企服务与分组输出

由于算法最终服务多个药企，输出结果必须按药企分组。

推荐输出：

```text
manufacturer_code
hospital_code
drug_group
cutoff_month
risk_H3
risk_H6
risk_H12
business_priority_H3
business_priority_H6
business_priority_H12
rank_in_manufacturer_H3
rank_in_manufacturer_H6
rank_in_manufacturer_H12
```

排序方式：

```text
先按 manufacturer_code 分组；
再在每个 manufacturer_code 内按 business_priority_score 排序；
输出每个药企自己的 TopK 风险清单。
```

这样可以充分利用全局模型学习到的共性，同时服务时仍然按药企隔离结果。

## 12. TopK 被少数产品占据的问题

可能出现：

```text
某药企的某几个高价值产品占据 TopK 大部分位置；
某些小产品线长期不进入 TopK。
```

这不一定是坏事，因为如果风险和价值确实集中，TopK 应当集中。

但为了产品可用性，可以额外输出两个版本：

### 12.1 全局价值优先 TopK

```text
按 business_priority_score 排序
不设产品线配额
```

适合管理层看最大风险敞口。

### 12.2 分产品线 TopK

```text
每个 product_line 输出 TopK
```

适合产品经理或区域负责人看覆盖面。

### 12.3 多样性约束 TopK

后续可考虑：

```text
同一 drug_group 最多占 TopK 的 X%
同一地区最多占 TopK 的 Y%
```

但 v1 不建议强加复杂约束，先观察排序结果分布。

## 16. 当前阶段结论

当前理解可以成立：

```text
1. 以医院×药品/产品线为分析单元；
2. 先按月分桶形成统一事实表；
3. 训练样本是 entity×cutoff；
4. 模型是全局模型，不是每个 entity 独立模型；
5. 目标从预测下次采购时间转为推荐高流失风险对象；
6. 指标以 Precision@K_H、NDCG@K_H、Lift@K_H 等排序指标为主；
7. 输出时按药企 manufacturer_code 分组，分别提供 TopK 风险清单；
8. 路由、动态窗口、BG/NBD、联邦/迁移学习暂不进入 v1，但数据结构应预留。
```
