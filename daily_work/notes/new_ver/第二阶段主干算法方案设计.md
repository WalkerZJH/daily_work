## 一、任务背景

当前 `BS_Agent_DingDan` 数据清洗层已经基本完成，并已能够读取较大规模 parquet 数据。接下来进入第二阶段：设计“终端不丢 / 用户存活”主干算法。

本阶段重点不是直接追求最终模型效果，而是先完成稳定、可回测、可扩展、可解释的算法设计框架，使后续可以逐步进行：

```text
model_base
→ 采购事实表
→ 医院×药品/产品线×时间切片
→ entity×cutoff 训练样本
→ 多窗口标签
→ 全局概率模型
→ 流失概率
→ 潜在损失估计
→ 业务优先级排序
→ 药企内 TopK 风险清单
→ metrics 报告
```

当前阶段不得推进前端、后端、工单系统、detector、Agent、全量数据库读取或复杂生产调度系统。

---

## 二、总体主线

当前 v1 最稳妥的主线是：

```text
v1 主干：基于流失概率的全局风险推荐模型
v1 输出：流失概率 + 潜在损失 + 业务优先级排序
v1 metrics：概率质量指标 + 多窗口 TopK 排序指标
v2 扩展：生存模型 / BG-NBD / Pareto-NBD 作为二阶段分析或特征来源
```

不要第一版就做：

```text
每个 医院×药品 entity 独立训练一个模型
```

推荐第一版采用：

```text
所有 entity×cutoff 样本共用一个全局模型；
每个 entity 通过自己的历史采购序列特征体现差异；
后续按药企、药品类别、需求形态、医院等级做分组评估和校准。
```

---

## 三、必须拆分的三个分数

本任务明确需要输出“流失概率”，因此主模型必须先预测纯粹概率，不能把价值直接混入概率定义。

必须拆分：

```text
churn_probability_H = P(die_H = 1)

value_at_risk_H = 如果该 entity 流失，可能造成的潜在采购损失

business_priority_H = churn_probability_H × value_at_risk_H × optional_weights
```

三者含义：

|字段|含义|是否可解释为概率|
|---|---|---|
|`churn_probability_H`|未来 H 窗口内停止采购的概率|是|
|`value_at_risk_H`|如果流失，可能损失的采购价值|否|
|`business_priority_H`|业务上是否值得优先处理|否|

Codex 必须遵守：

```text
1. 主模型只预测 churn_probability_H。
2. 不得把 business_priority_H 命名为 risk_score 或 churn_probability。
3. 不得把 die_H × value_at_risk_H 作为主概率模型目标。
4. 潜在损失和业务优先级必须作为后处理或二阶段排序策略。
```

## 四、第一版问题定义

第一版业务任务定义为：

```text
在每个 cutoff 月，对每个 药企 × 医院 × 药品/产品线 单元，
预测其未来 H 窗口内停止采购的概率，
并基于流失概率、潜在损失和业务优先级生成 TopK 风险清单。
```

第一版核心输出包括：


```text

```

## 五、第一版 entity 粒度

第一版推荐主分析单元：

```text
entity = manufacturer_code × hospital_code × drug_code
```

原因：

1. 系统最终需要服务药企，因此 `manufacturer_code` 必须贯穿全流程。
2. 当前没有真实 `product_line_code`，不要假设 `drug_category_code` 等价于产品线。
3. `drug_code` 粒度足够细，后续可以上卷到产品线或药品类别。
4. 输出时可以自然按 `manufacturer_code` 做药企内 TopK。

保留备选粒度：

```text
manufacturer_code × hospital_code × drug_category_code
hospital_code × drug_code
hospital_code × drug_category_code
hospital_code × product_line_code
manufacturer_code × hospital_code × product_line_code
```

v1 可以先实现：

```text
primary_entity = manufacturer_code × hospital_code × drug_code
secondary_entity = manufacturer_code × hospital_code × drug_category_code
```

用于后续对比粒度稳定性，但不要一次性把所有 entity 粒度都做成复杂主流程。

## 六、第一版采购事件和标签定义

### 6.1 采购事件

第一版采购事件定义：

```text
purchase_event = purchase_time 非空 且 order_detail_id 有效
```

只要发生采购事件，就认为该 entity 在该窗口内仍然活跃。

第一版不要因为以下状态否定采购事件：

```text
拒绝配送
无法配送
配送失败
未到货
退货
```

这些状态只能作为历史特征、履约解释或二阶段分析依据，不作为 v1 alive/die 标签的决定条件。

### 6.2 alive / die 标签

对每个 cutoff 和预测窗口 H：

```text
alive_H = cutoff 后 H 窗口内存在 purchase_event
die_H = 1 - alive_H
```

输出标签：

```text
label_alive_H3
label_die_H3

label_alive_H6
label_die_H6

label_alive_H12
label_die_H12
```

### 6.3 第一版 H 窗口

第一版必须同时支持：

```text
H = 3个月
H = 6个月
H = 12个月
```

含义：

|H|作用|
|---|---|
|H3|短期预警能力|
|H6|中期断购能力|
|H12|覆盖季节性、低频采购、长周期采购|

暂不把动态窗口作为主标签。动态 H、自适应 H、按需求形态设置 H，作为 v2/v3 研究方向保留接口即可。

## 十、潜在损失估计

第一版潜在损失不通过主概率模型直接学习，而是作为后处理估计。

### 10.1 金额口径

```text
value_at_risk_amount_H
= cutoff 前 12 个月平均月采购金额 × H
```

说明：

```text
金额字段是脱敏相对金额；
可用于相对排序、趋势和价值权重；
不能解释为真实货币金额；
不能用 金额 / 数量 推断真实单价。
```

### 10.2 数量口径

```text
value_at_risk_quantity_H
= cutoff 前 12 个月平均月采购数量 × H
```

说明：

```text
数量口径更稳；
但不同药品数量不可直接横向比较；
更适合同药品、同产品线、同药企内部比较。
```

### 10.3 第一版业务优先级

第一版简化：

```text
business_priority_score_H
= churn_probability_H × value_at_risk_amount_H
```

后续可扩展：

```text
business_priority_score_H
= churn_probability_H
  × value_at_risk_H
  × urgency_weight_H
  × confidence_weight
  × recoverability_weight
```

Codex 不要押注单一算法。必须保留多条候选路线。

## 十三、验证方式

不得使用随机 K-fold 作为主验证。

必须使用：

```text
rolling-origin temporal split
```

示例：

```text
Fold 1:
train cutoff <= 2022-12
valid cutoff = 2023-01 ~ 2023-03

Fold 2:
train cutoff <= 2023-03
valid cutoff = 2023-04 ~ 2023-06

Fold 3:
train cutoff <= 2023-06
valid cutoff = 2023-07 ~ 2023-09
```

对于不同 H，必须分别做 purged split：

```text
max_train_cutoff + H <= valid_start
```

同一个 entity 可以同时出现在 train 和 test，但必须时间方向正确：

```text
允许：A医院×B药品×2022-12 在 train，A医院×B药品×2024-06 在 test
禁止：A医院×B药品×2024-06 在 train，A医院×B药品×2022-12 在 test
```

普通 entity holdout / hospital holdout / drug holdout 只能作为辅助泛化测试，不作为主验证。

## 第二轮：算法框架对比

本轮目标是比较不同“风险推荐打分器”和二阶段备选模型，而不是把任务改成普通分类问题。

需要明确：

```text
本项目的业务形态是 TopK 风险推荐 / 风险排序；
但由于需求要求输出流失概率，v1 第一层主模型采用可校准的流失概率打分器；
概率模型输出的 churn_probability_H 用于回答“谁最可能流失”；
随后再基于 churn_probability_H × value_at_risk_H 计算业务优先级，用于回答“谁最应该优先处理”。
```

因此，本节中的“概率模型”不是指普通分类任务，而是指：

```text
推荐任务中的第一层概率打分器。
```

它的输出仍然会用于 TopK 排序：

```text
rank_by_probability_H = 按 churn_probability_H 降序排序
rank_by_business_priority_H = 按 churn_probability_H × value_at_risk_H 降序排序
```

统一默认参数先做横向比较，不要先做大规模超参数实验。

|框架类别|模型 / 方法|主要输出|定位|是否可解释为流失概率|
|---|---|---|---|---|
|规则 baseline|interval overdue / recent drop|`rule_score`|最低基线，用于验证机器学习是否超过简单规则|否|
|概率打分器 baseline|Logistic Regression|`churn_probability_H`|可解释概率基线，也是推荐排序的基础打分器|是|
|概率打分器主力候选|LightGBM|`churn_probability_H`|主力候选，适合大量统计特征|经校准后可以|
|概率打分器主力候选|CatBoost|`churn_probability_H`|高基数字段候选，适合 `hospital_code`、`drug_code`、`manufacturer_code` 等类别字段|经校准后可以|
|概率打分器稳定对照|XGBoost|`churn_probability_H`|稳定对照模型|经校准后可以|
|二阶段存活 / 间隔模型|BG/NBD、Pareto/NBD、采购间隔模型|`P_alive`、`expected_next_purchase_days`、`overdue_ratio`|二阶段解释、baseline 或特征来源|`P_alive` 可解释为存活概率，但口径不同于 `churn_probability_H`|
|排序 / rerank 模型|LightGBM Ranker、XGBoost Ranker、CatBoost Ranking|`ranking_score`|TopK 专用候选或二阶段 rerank，不作为 v1 唯一主线|否，除非额外校准|

本轮比较时必须分别评估两类指标：

### 1. 概率模型指标

用于评价 `churn_probability_H` 是否可靠：

```text
BrierScore_H
LogLoss_H
CalibrationCurve_H
ECE_H
AUC_H
PR_AUC_H
Precision@K_H by churn_probability
NDCG@K_H by churn_probability
Lift@K_H by churn_probability
```

这组指标用于回答：

```text
模型是否能可靠预测谁最可能流失？
```

### 2. 业务排序指标

用于评价加入潜在损失后的业务优先级是否合理：

```text
CapturedValue@K_H
ValueWeightedNDCG@K_H
ExpectedLossCaptured@K_H
TopK total value_at_risk_H
Precision@K_H by business_priority as guardrail
```

这组指标用于回答：

```text
在概率模型基础上，业务优先级排序是否能捕获更多高价值风险对象？
```

注意：

```text
1. churn_probability_H 是概率输出。
2. business_priority_score_H 不是概率。
3. 不得把 business_priority_score_H 解释为流失概率。
4. 不得用 ranking_score 替代 churn_probability_H，除非后续专门做概率校准。
5. 本项目的主任务仍然是 TopK 风险推荐，只是 v1 推荐打分器优先采用可校准的概率模型。
```

按特征组增量加入：

```text
A. recency/frequency 基础特征

C. + 采购间隔特征
D. + 状态历史特征
E. + 需求形态特征
```

每组都输出：

```text

NDCG@K_H
Lift@K_H

```

目标是回答：

```text
哪些信号真正有效；
哪些特征组会造成不稳定；
哪些特征组只提升业务价值排序但不提升概率质量。
```
