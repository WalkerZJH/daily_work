# 终端风险线索系统前后端模块设计书 v0.2

> 基准方案：B 方案——每日 as-of 特征重算 + 使用当前已验收主干模型进行日级评分 + 产出日报工作清单。  
> 当前定位：风险线索展示与业务复核系统。  
> 非当前定位：算法实验看板、模型训练配置平台、自动派单系统、客户可见模型参数平台。

---

## 0. v0.2 相对 v0.1 的核心调整

v0.1 已经明确了前后端的核心目标：算法产物进入后端、后端组织风险结果/风险卡/日报/proof-case/工单预留、前端展示业务人员可理解的信息、Parquet 到 ClickHouse 的平滑切换、权限和多租户预留等。

v0.2 在此基础上做四点修正：

1. **直接采用 B 方案作为日报基准**：每天根据最新订单数据重算 as-of 特征，再调用已验收的主干模型进行日级评分，生成每日风险线索与日报。
2. **区分训练批次、评分批次、日报批次**：训练不必每日执行；日报可以每日生成。
3. **前端展示对象从“模型结果”收敛为“工作清单”**：风险实体、风险卡、证据、趋势、proof-case、日报是业务端核心；训练参数、特征消融、AUC/ECE/PR-AUC 等默认不进入客户业务端。
4. **强化语义边界**：风险概率、业务优先级、survival 状态、detector 证据、one-shot 关注必须分开，不得互相替代。

## 1. 设计目标

本设计书不重新设计算法，也不直接实现前后端，而是明确：

1. 算法批处理产物如何进入后端；
2. 后端如何组织风险实体、风险卡、证据、日报、proof-case 和预留工单；
3. 前端如何展示业务人员能理解、能复核、能处理的信息；
4. 当前 Parquet 结果表如何在未来平滑切换到 ClickHouse；
5. 当前 mock 权限如何演进为严格多租户、多角色、多层级权限；
6. 哪些模块当前实现，哪些模块只预留接口；
7. B 方案下，日报如何在不每日重新训练模型的情况下生成。

系统当前应定位为：

- 风险线索展示系统；
- 业务复核系统；
- 日报工作清单系统；
- proof-case 历史案例展示系统。

系统当前不应定位为：

- 算法实验看板；
- 模型指标展示平台；
- 自动派单系统；
- 客户可见模型配置平台；
- 在线实时训练系统。

## 2. B 方案基准：训练层与日报层分离

### 2.1 B 方案定义

B 方案定义为：

**每日根据最新订单数据重算 as-of 特征，并使用已训练主干模型进行日级评分，生成当天风险线索和日报。**

这意味着：

| 层级 | 粒度 | 说明 |
|---|---|---|
| 模型训练层 | 月度或阶段性 | 用历史 cutoff 训练、回测、调参，不要求每日训练 |
| 日级评分层 | 每日 | 以 report_date 为 as-of 日期重算特征，调用当前有效模型输出风险概率 |
| 证据刷新层 | 每日 | 更新 interval、frequency、detector、是否恢复采购、是否持续风险 |
| 展示层 | 每日 | 生成企业内工作清单、风险卡、日报、历史案例入口 |

因此，产品“日报”不要求模型每日重新训练。日报依赖的是：

1. 最新订单数据；
2. 已验收模型；
3. 每日 as-of 特征；
4. 风险状态与证据刷新；
5. 工作清单生成规则。

### 2.2 关键时间字段

B 方案必须把以下时间字段分开：

| 字段 | 含义 |
|---|---|
| `report_date` | 日报日期，也是当日生成工作清单的业务日期 |
| `data_watermark_at` | 当前数据水位，表示订单数据同步到哪个时间点 |
| `score_as_of_date` | 日级评分使用的 as-of 日期，通常等于或早于 `data_watermark_at` |
| `training_cutoff_month` | 模型训练所用 cutoff 月份范围，不直接等于日报日期 |
| `model_batch_id` | 当前使用的模型批次 |
| `score_batch_id` | 当日评分批次 |
| `daily_report_id` | 当日日报批次 |

### 2.3 B 方案的优点

1. 满足日报需求；
2. 不破坏当前月度训练与回测体系；
3. 可以每天更新风险状态、采购恢复、证据变化；
4. 可以避免日级训练导致的样本高度重复和标签重叠；
5. 便于后续从 Parquet 切换到 ClickHouse；
6. 便于接入定时任务和日报推送。

### 2.4 当前不采用的方案

当前不采用：

1. 纯月度结果直接展示，不更新每日状态；
2. 日级 cutoff 重新训练主模型；
3. 实时在线预测；
4. 每次用户打开页面才触发模型计算；
5. 让前端直接读取算法实验输出。

## 4. 用户角色与权限模型

### 4.1 当前最小角色

| 角色 | 当前是否实现 | 说明 |
|---|---:|---|
| `platform_admin` | 可 mock | 平台内部管理员，维护批次、租户、导入配置 |
| `tenant_admin` | 预留 | 客户企业管理员，管理本企业用户和范围 |
| `region_manager` | 当前核心 | 查看本区域风险实体、风险卡、日报、proof-case；未来可创建工单 |
| `sales_rep` | 预留 / 快速切换 | 未来查看分配给自己的工单，提交处理反馈 |
| `senior_manager` | 预留 | 查看下级经理数据和风险结果 |
| `readonly_viewer` | 预留 | 只读查看日报、proof-case、风险结果 |

### 4.2 tenant 与 manufacturer 的关系

不能把 `tenant_id` 简单等同于 `manufacturer_code`。

建议：

| 字段 | 含义 |
|---|---|
| `tenant_id` | 客户企业或客户组织账号 |
| `enterprise_id` | 企业主体 |
| `manufacturer_code` | 具体生产商或生产厂家编码 |

关系：

1. 一个 `tenant` 一般绑定一个 `enterprise`；
2. 一个 `enterprise` 可以绑定一个或多个 `manufacturer_code`；
3. 风险计算当前粒度仍以 `manufacturer_code` 为主；
4. 业务权限必须从 `tenant_scope` 推导到 `manufacturer_scope`。

### 4.3 权限过滤原则

所有 API 必须经过权限过滤：

用户登录态 → tenant_id → role → data_scope → allowed_enterprise_ids → allowed_manufacturer_codes → allowed_region_codes → 查询风险结果。

前端传入的 `manufacturer_code`、`region_code`、`hospital_code` 只能作为筛选条件，不能作为权限依据。后端必须强制取交集：

`query_scope = frontend_filter ∩ user_permission_scope`

## 5. 核心业务对象

### 5.1 RiskEntity 风险实体

风险实体是业务处理的核心对象。

当前粒度：

`manufacturer_code × hospital_code × drug_code × score_as_of_date`

其中：

1. `horizon` 是风险观察窗口，不一定构成实体主键；
2. 一个实体可以同时有 H3/H6/H12 三组风险结果；
3. 当前 `drug_group_source = drug_code`；
4. 未来可扩展为 `manufacturer_code × hospital_code × product_line_code × score_as_of_date`。

当前应预留：

| 字段 | 说明 |
|---|---|
| `product_line_code` | nullable，未来产品线映射 |
| `drug_group_source` | 当前为 `drug_code` |
| `score_as_of_date` | B 方案下的日级评分日期 |
| `report_date` | 日报展示日期 |

### 5.2 RiskCard 风险卡

风险卡不是实体本身。一个风险实体可以对应多个风险卡。

示例：

1. 主风险卡；
2. H3 / H6 / H12 风险卡；
3. interval 超期证据卡；
4. 采购频次下降证据卡；
5. proof-case 关联卡；
6. 经理手动备注卡。

关系：

`RiskEntity 1 : N RiskCard`

### 5.3 RiskEvidence 风险证据

证据是风险卡的可解释材料。

证据示例：

1. 距离上次采购时间明显长于历史常规节奏；
2. 近 3 个月采购频次低于过去 12 个月水平；
3. 该医院对本生产商其他药品采购也出现下降；
4. 该药品在本生产商内整体采购活跃度下降；
5. 历史相似关系中，该类状态后续停购比例较高。

证据不等于概率，不得把 detector severity 或 survival confidence 展示为概率。

### 5.4 DailyReport 日报

日报是业务端可见的正式报告对象。

日报内容：

1. `report_date`；
2. 当前使用的 `score_batch_id`；
3. 数据水位 `data_watermark_at`；
4. 今日新增高风险实体数；
5. 今日持续高风险实体数；
6. 今日风险缓解实体数；
7. 重点医院数；
8. 重点药品数；
9. 重点风险实体列表；
10. proof-case 摘要；
11. 系统建议关注方向；
12. 往期日报入口。

日报不复制全部风险数据，而是引用风险实体、风险卡和 proof-case。

### 5.5 ProofCase 历史命中案例

ProofCase 是业务端可见的历史命中案例，不是算法指标页。

展示内容：

1. 过去某个 cutoff 或 report_date 系统判断为高风险；
2. 后续真实发生停购或风险兑现；
3. 当时主要业务证据；
4. 相关采购趋势；
5. 业务人员可理解的命中说明。

不展示：AUC、ECE、PR-AUC gain、LogLoss、特征消融、模型训练参数。

### 5.6 WorkOrder 工单

当前不实现完整工单，但必须预留结构。

建议流程：

1. 经理查看某个 RiskEntity；
2. 选择一个或多个 RiskCard；
3. 点击“生成工单”；
4. 系统创建 WorkOrder；
5. WorkOrder 绑定 RiskEntity，并引用多个 RiskCard。

关系：

1. `RiskEntity 1 : N WorkOrder`；
2. `WorkOrder N : M RiskCard`。

## 6. 批次与数据流设计

### 6.1 总体链路

算法批处理结果 → Parquet 结果批次 / 未来 ClickHouse 结果表 → Repository 抽象层 → 后端服务 → 业务端 Vue 页面。

### 6.2 B 方案下的批次类型

| 批次 | 说明 |
|---|---|
| `ModelBatch` | 模型训练批次，包含训练窗口、模型版本、评估摘要 |
| `ScoreBatch` | 日级评分批次，包含 report_date、score_as_of_date、data_watermark_at |
| `RiskResultBatch` | 风险实体、风险卡、证据、聚合和日报的标准输出批次 |
| `DailyReportBatch` | 日报输出批次，可与 RiskResultBatch 同批或独立生成 |
| `ProofCaseBatch` | 历史命中案例批次 |

### 6.3 标准输出目录

当前 Parquet 结果建议目录：

`outputs/risk_result_batches/batch_id=<score_batch_id>/`

包含：

1. `manifest.json`；
2. `risk_entities.parquet`；
3. `risk_cards.parquet`；
4. `risk_card_evidence.parquet`；
5. `risk_entity_timeline.parquet`；
6. `hospital_aggregates.parquet`；
7. `drug_aggregates.parquet`；
8. `proof_cases.parquet`；
9. `daily_reports.parquet`；
10. `work_order_reserved.parquet`，可选。

### 6.4 manifest.json

`manifest.json` 应至少包含：

| 字段 | 说明 |
|---|---|
| `batch_id` | 风险结果批次 ID |
| `schema_version` | 结果 schema 版本 |
| `model_batch_id` | 使用的模型批次 |
| `score_batch_id` | 日级评分批次 |
| `report_date` | 日报日期 |
| `score_as_of_date` | 日级特征 as-of 日期 |
| `data_watermark_at` | 数据水位 |
| `available_horizons` | H3/H6/H12 |
| `default_horizon` | 默认展示 horizon |
| `data_backend` | parquet / clickhouse |
| `created_at` | 批次生成时间 |
| `service_level` | internal_diagnostic / analyst_view / customer_facing |
| `auto_dispatch_allowed` | 当前固定 false |

## 11. 风险颜色和概率展示规则

概率可展示，但必须受 `probability_display_allowed` 和 `display_mode` 控制。

示例：

| display_mode | 展示方式 |
|---|---|
| `show_probability` | 展示概率和风险等级 |

| `hide_probability` | 显示历史不足或数据不足 |

风险颜色示例配置：

| 类型 | 颜色 | 说明 |
|---|---|---|
| high | 红色 | 高风险，建议优先人工复核 |
| medium | 橙色 | 中风险，建议关注 |
| low | 蓝/绿 | 低风险或一般对象 |
| observation_only | 灰色 | 观察对象，不代表低风险 |
| history_insufficient | 灰色 | 历史不足，不适合强概率结论 |

## 21. 最终设计结论

当前前后端应围绕以下主线设计：

1. 算法按 B 方案产生日级评分批次和日报批次；
2. 后端以标准结果表为中心提供查询、聚合、日报、proof-case 和工单预留能力；
3. 前端以业务人员能理解的风险概率、风险等级、证据、采购趋势和历史命中案例为中心展示；
4. 模型、特征、训练细节不作为客户业务端核心内容；
5. 权限、租户隔离、工单、ClickHouse、产品线粒度必须在接口和数据模型中预留；
6. B 方案下，日报每天生成，但模型不需要每天重新训练；
7. 当前系统仍应保持 `auto_dispatch_allowed=false`，所有高风险线索默认需要人工复核。

推荐架构：

算法日级评分批处理结果  
→ Parquet 结果表 / 未来 ClickHouse 结果表  
→ Repository 抽象层  
→ 风险实体服务 / 风险卡服务 / 日报服务 / Proof-case 服务 / 工单预留服务  
→ 业务端 Vue 页面

这套设计可以避免当前被复杂前后端实现拖住，同时保证后续需求部门提出工单、权限、日报、聚合视图、客户隔离、ClickHouse 接入时，不需要推翻已有结构。
