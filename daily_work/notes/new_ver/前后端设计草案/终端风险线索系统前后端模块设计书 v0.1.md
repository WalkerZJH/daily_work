## 1. 设计目标

本设计书的目标不是重新设计算法，也不是直接实现前后端，而是明确：

```text
1. 算法产物如何进入后端；
2. 后端如何组织风险结果、风险卡、日报、回测案例和预留工单；
3. 前端如何面向业务人员展示可理解、可行动的信息；
4. 当前 Parquet 结果表如何在未来平滑切换到 ClickHouse；
5. 当前简单权限如何演进为严格多租户、多角色、多层级权限；
6. 哪些模块当前实现，哪些模块仅预留接口。
```

系统当前应定位为：

```text
风险线索展示与业务复核系统
```

而不是：

```text
算法实验看板
模型指标展示平台
自动派单系统
客户可见模型配置平台
```

算法层已经可以输出较高质量的风险概率、风险等级、证据和历史命中案例；前后端需要做的是把这些内容组织成业务人员能直接理解和处理的页面。

## 4. 租户隔离与数据安全

### 4.1 tenant 与 manufacturer 的关系

由于一个企业可能绑定多个生产商，不能简单把 `tenant_id` 等同于 `manufacturer_code`。

建议设计为：

```text
tenant_id = 客户企业 / 客户组织账号
enterprise_code = 企业主体
manufacturer_code = 具体生产商 / 生产厂家编码
```

关系：

```text
一个 tenant 一般只绑定一个 enterprise，在目前开发层面，这个论断是肯定的；
一个 enterprise 可以绑定一个或多个 manufacturer_code(也可以只绑定一个)；
风险计算当前粒度仍以 manufacturer_code 为主；
业务权限必须从 tenant_scope 推导出 manufacturer_scope。
```

### 4.2 数据隔离原则

所有 API 都必须经过权限过滤：

```text
用户登录态
→ tenant_id
→ role
→ data_scope
→ allowed_enterprise_ids
→ allowed_manufacturer_codes
→ allowed_region_codes
→ 查询风险结果
```

前端传入的 `manufacturer_code`、`region_code`、`hospital_code` 只能作为筛选条件，不能作为权限依据。后端必须强制取交集：

```text
query_scope = frontend_filter ∩ user_permission_scope
```

### 4.3 当前阶段处理方式

当前没有真实权限系统时，也应该在接口和数据结构中保留：

```text
tenant_id
enterprise_id
manufacturer_code
region_code
owner_manager_id
assigned_sales_rep_id
```

即使当前这些字段暂时写固定值，也要避免后续重构数据模型。

## 5. 核心业务对象

### 5.1 风险实体 RiskEntity

风险实体是业务处理的核心对象。根据需求侧，一个企业一期、一天最多处理50个风险对象，倾向于默认20个风险对象。由于目前的主干算法是基于整体的topK，所以可能存在部分生产企业某期不存在风险对象，需要考虑使用混合排序办法。

当前粒度：

```text
manufacturer_code × hospital_code × drug_code × cutoff_month
```

其中 `horizon` 是风险概率的观察窗口，不一定构成实体主键。一个实体可以同时有 H3/H6/H12 三组风险结果。

未来可能扩展：

```text
manufacturer_code × hospital_code × product_line_code × cutoff_month
```

但当前由于生产线字段不存在，只预留：

```text
product_line_code nullable
drug_group_source = drug_code
```

### 5.2 风险卡 RiskCard

风险卡不是实体本身。一个风险实体可以对应多个风险卡。

例如同一个：

```text
某生产商 × 某医院 × 某药品
```

可能有多个风险卡：

```text
1. H3 高风险卡；
2. H6 中风险卡；
3. interval 超期证据卡；
4. 采购频次下降证据卡；
5. proof-case 关联卡；
6. 经理手动添加的备注卡。
```

因此关系是：

```text
RiskEntity 1 : N RiskCard
```

### 5.3 工单 WorkOrder

工单不是风险卡。

建议设计为：

```text
经理查看某个 RiskEntity；
选择一个或多个相关 RiskCard；
点击“生成工单”；
系统创建 WorkOrder；
WorkOrder 绑定 RiskEntity，并引用多个 RiskCard。
```

关系：

```text
RiskEntity 1 : N WorkOrder
WorkOrder N : M RiskCard
```

当前不实现工单，但应预留这套结构。

### 5.4 日报 DailyReport

日报是业务端可见的正式报告对象。

日报中可以包含：

```text
1. 今日新增高风险实体数；
2. 高风险医院数；
3. 高风险药品数；
4. 重点风险实体列表；
5. 已处理 / 待处理统计；
6. 往期命中案例摘要；
7. 简单业务化指标。
```

日报能展示的信息，往期日报也可以展示。

### 5.5 ProofCase

ProofCase 是历史命中案例，不是算法指标页。

它应展示：

```text
1. 过去某个 cutoff 时系统判断为高风险；
2. 后续真实发生停购 / 风险兑现；
3. 当时的主要业务证据；
4. 当前可解释的简化指标；
5. 业务人员可理解的命中说明。
```

可展示简单指标，但不展示模型训练细节。

## 8. 风险实体列表模块

### 8.1 列表主字段

风险实体列表展示的是实体级信息，而不是每条证据卡。

建议字段：

```text
risk_entity_id
企业 / 生产商
医院
药品
默认 horizon 风险概率
风险等级
风险颜色
最近一次采购日期
距离上次采购天数
历史采购节奏
主要风险原因
风险卡数量
是否有多个药品同院高风险
是否已关注
是否已生成工单
处理状态
```

### 8.2 风险颜色规则

概率可展示，并按概率区间上色。阈值不要硬编码，应走配置。

示例配置：

```text
high: probability >= 0.70，红色
medium: 0.40 <= probability < 0.70，橙色
low: probability < 0.40，蓝/绿
observation_only: 灰色
history_insufficient: 灰色
```

灰色含义：

```text
不代表低风险，而是当前历史不足或仅观察，不适合给强概率结论。
```

### 8.3 horizon 切换

默认只展示一个 horizon，例如 H3 或 H6。具体默认值走系统配置。

每张风险实体卡或风险详情中允许快速切换：

```text
H3 / H6 / H12
```

切换后更新：

```text
risk_probability
risk_level
risk_reason_summary
risk_cards
evidence
timeline interpretation
```

## 14. 工单系统预留设计

### 14.1 当前状态

当前不实现完整工单系统，但必须在对象模型和接口中预留。

### 14.2 工单创建逻辑

未来经理操作流程：

```text
1. 经理查看风险实体；
2. 选择一个或多个风险卡；
3. 点击“生成工单”；
4. 选择业务员或自动分配；
5. 业务员处理；
6. 业务员反馈；
7. 经理复核；
8. 工单关闭。
```

```text
work_order_id

created_by_manager_id
assigned_sales_rep_id
status
priority
created_at
updated_at
due_date
assignment_mode
```

### 14.4 工单与风险卡关联

```text
work_order_card_link:
  work_order_id
  risk_card_id
```

允许：

```text
一键添加该 entity 下所有相关风险卡；
手动选择部分风险卡；
后续追加风险卡。
```

### 14.5 自动分配预留

未来经理端支持：

```text
自动分配给当前闲置业务员；
均衡一段时期内业务员工作量；
避免同一业务员过载；
同一医院可优先分配给历史负责人。
```

预留字段：

```text
sales_rep_current_load
sales_rep_recent_load_7d
sales_rep_recent_load_30d
assignment_weight
assignment_rule_id
```

### 14.6 并发处理预留

工单未来会有并发操作，需要预留：

```text
version
locked_by
locked_at
last_operation_id
operation_log
```

后端更新时采用乐观锁：

```text
update where work_order_id = ? and version = ?
```

失败则提示前端刷新。

### 19.3 模型信息处理原则

模型版本和训练细节用于程序员管理，不直接进入客户业务端。

客户业务端可以看到：

```text

风险等级
风险原因
历史命中案例
业务化指标
往期日报
```

客户业务端不需要看到：

```text

训练参数
训练数据分割
特征工程细节
原始 SHAP 数值
模型消融结果
内部泄露审计报告
```

但如果需要展示结果质量，应转化为业务语言：

```text
历史高风险线索命中比例
高风险区域风险密度提升

```

## 24. 仍需后续确认的问题

这份设计已经可以作为第一版，但仍有几个后续应确认的问题：

```text
1. 默认 horizon 是 H3 还是 H6；
2. 风险等级概率阈值是否由需求部确认；
3. 业务化指标中是否允许展示“命中率”这个词；
4. 是否需要客户业务端展示“风险密度提升倍数”；
5. 风险卡是否允许经理手动新增；
6. 日报是系统自动生成，还是经理可编辑后发布；
7. 工单系统是否需要审批流；
8. 区域经理的数据范围来自医院区域、销售区域，还是企业内部组织树；
9. 一个医院跨多个区域时按什么规则归属；
10. 业务员是否只能看工单，还是也能看风险实体详情。
```

## 25. 最终设计结论

当前前后端应围绕这个主线设计：

```text
算法输出风险实体与风险卡；
后端以结果表为中心提供查询、聚合、日报、proof-case 和预留工单能力；
前端以业务人员能理解的风险概率、风险等级、证据、采购趋势和历史命中案例为中心展示；
模型、特征、训练细节不作为客户业务端核心内容；
权限、租户隔离、工单和 ClickHouse 接入必须在接口和数据模型中预留。
```

最推荐的架构是：

```text
算法批处理结果
  ↓
Parquet 结果表 / 未来 ClickHouse 结果表
  ↓
Repository 抽象层
  ↓
风险实体服务 / 风险卡服务 / 日报服务 / Proof-case 服务 / 工单预留服务
  ↓
业务端 Vue 页面
```

这套设计可以避免当前就被复杂前后端实现拖住，同时保证后续需求部提出工单、权限、日报、聚合视图、客户隔离时，不需要推翻已有结构。
