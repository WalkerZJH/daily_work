## 〇、核心框架转变（本版的灵魂）

**从滞后指标 → 领先指标。**

| | 旧框架：终端丢失检测 | 新框架：终端不丢 |
|---|---|---|
| 指标性质 | 滞后（已经丢了才报） | 领先（刚出现风险就报） |
| 产品比喻 | 验尸报告 | 体检预警 |
| 干预窗口 | 已关闭，挽回成本高、成功率低 | 仍开着，医院还在进货、关系还在 |
| 判定方式 | 二元（丢/没丢），易误报、易被一线否掉 | 多维加权风险分 + 预警等级 + 证据链，可解释、有分级 |

同样一份订单数据，把产品价值从"告诉你终端死了"变成"提前告诉你它在变虚"。一线能干预的预警，才是他真正肯天天看的。

## 一、问题陈述（Problem Statement）

站在**药品生产企业**视角，以**多维度算法巡检**持续扫描已采集的本企业产品订单数据，**及时发现"存在终端丢失风险"的早期线索**；通过**多维算法加权计算风险分**，定义**预警等级**，及时提醒企业客户；并为每条风险线索提供**可溯源的证据链**，让一线看得懂、信得过、能据此干预。

目标不是"报出已经丢的终端"，是"让终端不丢"。

## 五、前提（Premises）— 已确认，并按新框架更新

1. **单点切入**：MVP 只做"终端不丢（流失风险巡检预警）"，价格/配送/销量独立监测、驾驶舱先进 backlog。
2. **基线 = 生死线（已被新框架强化）**：不再做"丢/没丢"二元判定，改为**多维加权风险分**。每个维度仍需按「医院×药品（或产品线）」历史周期建基线消除正常波动，但最终对一线呈现的是连续风险分与等级，天然降低误报杀伤、提升可解释性。
3. **推给人优先 + 证据链**：价值主动送到对的人面前，且每条预警必须附**证据链**（哪几个维度触发、各自贡献多少分、对应哪些订单事实），否则一线不信、不用。工单闭环干预。管理层驾驶舱第二位。
4. **单客户验证**：先深度绑定已表达买单意愿的客户，把风险分的准确度与一线采纳率打磨好，再复制。

## 六、算法架构（本方案主体）

### 6.0 设计认知（经 Fader / Syntetos / Hyndman / Provost 视角审定）

不是做"一个判会不会丢的分类器"。明确**分工**：

- **分数交给统计干净的主干**：P(alive) 存活概率 ×（价值加权 / v2 uplift 可挽回性）。它可校准、可回测，是唯一进入"风险高低"的量。
- **探测器主要负责证据链 + 根因分流**（断供型/需求型/竞品型/政策型），**次要**负责"多族印证→升级"。它们不是一堆需要互相去相关的独立打分项——这避免了"7 探测器 Frankenstein"的统计陷阱（多重检验、相关污染）。
- **一线看的最终排序不是风险分，是"挽回价值"**（见 6.9）。风险分是输入，不是决策。

可解释证据链 + 印证逻辑是落地前提，也是前提2"生死线"的技术实现。

### 6.1 七层巡检流水线

```
L0  数据预处理与基线
L0.5 需求形态分类闸门（强制）  ← 先分类再探测，否则间断型药品=误报喷泉
L1  主干打分 + 多算法并行探测
L2  去伪校正(同侪 + 政策/集采)
L3  融合 + 交叉验证(印证升级)  +  FDR 多重检验控制
L4  分级 + 证据链 + 根因标签
L4.5 决策排序(挽回价值 = 可挽回性 × 终端价值 − 成本)
L5  处置反馈校准(含概率校准)
```

每天"巡检"= 这条流水线对全量「医院×产品线」组合跑一遍。

### 6.05 L0.5 需求形态分类闸门（强制前置 · Syntetos）

> Syntetos 砖："这不是脚注，是第一道闸门。没分类就上探测器，误报率压不下来，生死线当场失守。"

```
对每个分析单元 u 算两个量：
  ADI  = 平均需求间隔（有采购的月之间的平均间隔）
  CV²  = 采购量变异系数的平方
按 Syntetos-Boylan 四象限分类并路由：
  平稳(smooth)   ADI<1.32, CV²<0.49 → 全套探测器、标准容忍带
  波动(erratic)  ADI<1.32, CV²≥0.49 → 趋势类探测器降权，放宽量波动容忍
  间断(intermittent) ADI≥1.32, CV²<0.49 → 禁用 MK 趋势检验，IPI 用 Croston/SBA 期望补货点
  块状(lumpy)    ADI≥1.32, CV²≥0.49 → 仅保留 P(alive) 主干 + 期望补货点，关掉敏感探测器
```
**关键**：间断/块状序列上**禁止跑 Mann-Kendall 趋势检验**（方法论错误，会把随机性当衰减）；改用 Croston/SBA 估期望补货点，超期才报。

### 6.2 主干指标：存活概率 P(alive) ×（价值 / 可挽回性）

终端流失本质是营销科学的客户存活问题（buy-till-you-die，Fader-Hardie）。用 **BG/NBD 或 Pareto/NBD**：

- 输入每个医院的 **RF**：最近一次采购距今（Recency）、历史采购频次（Frequency）。
- 输出："仍是活跃买家"的概率 **P(alive)** 与"预计下次采购时间"。
- **P(alive) 下行趋势 = 最干净的领先指标**，把"节奏放缓"数学化成可校准的概率，避免硬阈值误报雷区。

> **Fader 砖（必须正视的假设边界）**：BG/NBD 假设非契约、独立泊松、只"死一次"的同质买家。**医院采购受处方集/集采/医保驱动，"死"常是政策事件而非随机 dropout。** 应对：① 政策性断崖交 L2 集采掩码处理，不让主干把它当自然流失；② BTYD 仅作骨架，最终分数乘价值/可挽回性；③ 主干参数按机构等级/采购形态簇分层拟合，缓解异质性。

**主干风险分** = (1 − P_alive)×100。**但它不是最终排序量**——见 6.9 决策层。

### 6.3 并行探测器（主职：证据 + 根因分流；次职：印证）

> 经审定，探测器的**首要价值是产出证据链与根因标签**，让一线看懂、分流处置；**次要**才作"独立信号族"参与印证升级（6.5）。它们不再是需要互相去相关的打分器——这正面回应 Fader 的"Frankenstein"砖与 Hyndman 的"多重检验"砖。

每个探测器输出：**命中标志 + 严重度 s∈[0,100] + 置信度 c∈[0,1] + 根因标签**（c 由样本量决定，样本不足 c→0 不参与）。

**第一版核心（v1）：**
| 方向 | 算法 | 命中信号 | 主要输入字段 |
|---|---|---|---|
| 1 采购节奏异常(IPI) | 历史采购间隔中位/MAD，鲁棒 z 分；间断型改 Croston/SBA 期望补货点 | 距上次下单天数超期望 | 下单时间、药品编码、医疗机构编码 |
| 2 量/金额趋势衰减 | Mann-Kendall + Theil-Sen + CUSUM（**仅平稳/波动型可用**） | 显著单调下行 / 持续均值下移 | 采购数量、采购金额、下单时间 |
| 3 采购频次衰减 | 泊松/负二项到达率，λ_recent vs λ_base | 近窗口下单率显著下降 | 订单编号、下单时间 |
| 4 品规收窄/钱包份额 | 活跃品规计数趋势；份额下行 | 在采品规数收缩（竞品逐个替代前兆） | 药品编码、规格、剂型 |

**第二版（backlog，依赖数据/算法成熟度）：**
| 方向 | 算法 | 命中信号 | 主要输入字段 |
|---|---|---|---|
| 5 履约三段差(断供前兆) | 采购→配送→收货三段缺口；配送率/时效趋势 | 配送率↓、配送时延↑、拒单率↑ | 采购/配送/收货数量·时间、配送企业 |
| 6 价格/竞争替代 | 同药品编码跨院/跨省价格离散度突变 | 出现异常低价/跨省价差且时间领先于量价下滑 | 采购价格、制剂价格、批准文号 |
| 7 同侪对照(去伪) | 按等级+地区分群，相对风险=本院下滑−同群中位下滑 | 群体稳而单院特异下滑 | 机构等级、所属地区 |

> **方向5 极重要**：三段差区分"医院想买但买不顺（配送商问题）"与"医院不想买了（需求流失）"——根因与干预动作完全不同，绝不能混判。

### 6.4 L2 去伪校正（融合前先扣掉"假流失"）

- **集采事件**：用 医保目录类型 + 量价阶跃突变识别集采中标/落标周期；落标导致的合法量跌打掩码、不计风险。
- **季节/周期**：STL 分解或同比同期，扣除季末囤货、节假日。
- **配送商系统性失败**：同一配送企业服务的一批医院同时履约恶化 → 单独成"修配送商"线索，不摊到每个医院流失风险上。

### 6.5 L3 融合 + 交叉验证 + FDR 控制（核心）

**(1) 分数以主干为准，探测器只做印证升级**
```
风险分 Score = 主干分(1−P_alive)×100 × 印证系数
印证系数 = 1 + α·(独立命中信号族数 k − 1)   // 封顶
```
- 探测器不直接加权进分数主体（避免相关污染/多重检验），只通过"是否多族印证"影响**分级**。
- 概率派可选：将各探测器校准为"该维度指示流失的概率"，用 **Noisy-OR** 融合做印证。

**(2) FDR 多重检验控制（Hyndman 砖，必做）**
> 每天对数千「医院×产品线」×多探测器跑检验，按 p<0.1 纯随机也会"发现"成百上千条假风险。
- 对所有探测器的显著性检验，用 **Benjamini-Hochberg 控制错误发现率（FDR）**，目标 FDR 默认 ≤ 10%，而非每个探测器各自拍 p 值。
- 经 FDR 过滤后存活的命中，才算"独立命中信号族"，才参与印证升级。

**(3) 模型交叉验证（权重不靠拍脑袋）**
- v1 无标签：主干 + 印证系数起步，探测器走专家规则。
- 有回测标签后：可学一个**校准良好**的流失概率模型（逻辑回归/GBDT，特征含主干分与探测器输出）；用**时序滚动交叉验证(walk-forward CV)**（非随机 k 折，防时间穿越）；输出做**概率校准(Platt/Isotonic)**。**重判别力更重校准**——风险分 78 必须真对应约 78% 概率，否则分级阈值无意义。详见附录 C。

### 6.6 L4 预警分级（印证逻辑驱动）

- **红（高危）**：P_alive 低 **且 ≥2 个独立信号族（经 FDR 过滤）**命中 **且** 未被 L2 解释 → 挽回窗口收窄。
- **橙（关注）**：P_alive 中等 **或** 1 个强信号族 + 同侪对照确认特异 → 本周跟进。
- **黄（观察）**：单一信号族命中 / 苗头初现 → 入观察清单。
- 阈值可配置，由回测校准（基于校准后概率，非原始判别分）。

> 分级回答"风险多高"。但**一线该先打哪个电话，由 6.9 决策排序决定，不是直接按风险分排序。**

### 6.9 L4.5 决策层：从"风险分"到"挽回价值"（Provost + Fader）

> Provost 砖："你建的是'谁会流失'，但业务问题是'我联系谁才能改变结果'。" Fader 砖："不是所有终端一样值钱。"

风险分只是输入。一线工单的**排序量**应是**期望挽回价值**：

```
挽回价值(u) = P(可挽回 | 干预)  ×  终端价值 CLV(u)  −  干预成本
```

- **终端价值分层（v1 即可上）**：用历史采购金额 + 机构等级 + 预测 CLV 给每个终端打价值档（高/中/低）。**风险分 × 价值档** 决定优先级——别让一线把精力砸在挽不回的小户上。
- **可挽回性 / uplift（v2）**：真正的目标不是流失概率，是"被干预后留存概率的提升"。区分三类终端：①已铁心走（干预无效）②本会留（干预浪费）③**可被推回的 persuadables**（真正该投精力）。落地：待积累足够"干预-结果"配对数据后，训练 uplift 模型（two-model / 因果森林），把排序从"风险高"升级为"可挽回价值高"。
- **护栏（Provost）**：证据链是采纳关键，但**不能让解释沦为对错误分数的事后合理化**——解释一个错的预测比没有解释更危险。低置信/低校准的线索须显式标注，不强行编因果。

### 6.7 证据链设计

每分来源都能追到具体订单事实，回答一线三问：**为什么判有风险 / 风险多高 / 我看哪些事实**。

```
风险线索：XX三甲医院 · 本企业A产品线 · 风险分 78（橙→红临界）· P(alive) 0.34↓
触发探测器与贡献：
  · 节奏异常(IPI)  +32   近3次采购间隔 28→41→63 天，历史中位 30 天
  · 量趋势衰减     +24   月采购量近3月 1200→900→520 盒，MK 检验显著下行
  · 品规收窄       +22   在采品规由 4 个降至 1 个
去伪校正：同群三甲医院本期采购稳定，本院属特异下滑（非集采/季节）
建议动作：区域经理2日内回访，核实是否竞品替代/院内政策变动
证据可溯源至订单：[订单号清单]
```

- **护栏**：禁止无依据加分；探测器数据不足记"样本不足/不计分"，不臆测。

### 6.8 两个工程现实（生死线落地）

1. **判定单元**：主风险分建在「医院×产品线」（稳），下钻证据用「医院×品规」（细），品规级信号上卷到产品线级。
2. **冷启动**：历史 < N 单时用**同侪先验做贝叶斯收缩**（借同群典型 IPI/量价画像当先验），标低置信度，宁可不报不瞎报。

> 需求形态分类（Syntetos-Boylan + Croston/SBA）已提升为强制前置环节 L0.5，见 6.05。

## 七、智能体角色（AI 在本方案中做什么）

AI 不替代风险分（风险分必须是可复现、可解释、可校准的算法，必须稳），AI 做两件加分的事：
1. **证据链自然语言化**：把探测器命中事实翻译成一线一眼能懂的一句话诊断 + 建议动作。
2. **风险归因**（v2）：在红色预警上，用订单自带的配送/价格字段辅助判断风险类型（断供型/政策型/竞品替代型），低置信度标"待人工核实"。

关键：**算法巡检负责"稳和可解释"，AI 负责"省心和好懂"，算分不交给 AI，二者分工、互不背锅。** 解释不得沦为对错误分数的事后合理化（Provost 护栏）。

## 八、产品形态（Product Form）— 经几轮沟通逐条敲定

> 算法回答"哪些终端在变虚"。本章回答"它做成一个什么东西、谁在用、怎么用、先做哪一截"。算法再准，产品形态错了照样进坟场。

### 8.0 一句话形态判断

**不是 BI 分析工具，是「风险线索的工作流收件箱」。** 算法每天巡检 → 生成会说人话的线索卡 → 按挽回价值排序 → 推到对的人面前 → 像处理 CRM 待办一样逐条处置 → 处置反馈回流校准算法。要躲开的就是"一块没人看的漂亮仪表盘"。

### 8.1 产品架构：独立产品 + 数据层接入 + 入口桥接（地基）

- **产品层：独立**。不做现有 SaaS 里的内嵌模块，做一个**独立的 SaaS 端智能体服务产品**——自有 UI / 品牌 / 完整设计自由度（每日屏、转发流、状态机都需要老 SaaS"基础预警查看"界面给不了的交互）。便于打包、定价、独立迭代，作为单独 SKU 卖给 200+ 药企。
- **数据层：接入（寄生在数据上）**。后台只读接入现有供应链 SaaS 的底层订单数据源。归因所需字段本就在同一条订单记录里（核心 EUREKA 不变）。巡检日频 → **每日批量增量同步即可，无需实时**，工程很轻。
- **入口桥接（关键，别丢分发红利）**。独立产品的代价是"VP 多一次登录"。两道桥消除：① 从现有 SaaS 一键 SSO 跳入；② 企微晨报 deeplink 直达某条线索。让 VP 感觉不到"第二个系统"。
- **口诀：数据从老系统来，产品独立站，报警往企微去。**

> **数据接入契约别写死**：v1 单客户可直连底层库（只读副本/视图）图快，但取数中间隔一层**标准字段 schema**，否则铺开到 200 家、每家库结构不同时要重写。需确认历史深度（base 窗需 91–365 天、回测需 12 个月+）。多租户隔离（每家只看自己数据）架构上先留口子。

### 8.2 产品原子 = 线索卡

用户每天触摸的最小单位是**一张线索卡**：医院 × 产品线 × 风险等级 × P(alive) × 证据链三行 × 建议动作 × 处置状态。整个一线产品 = 一个线索卡的队列/收件箱。

**风险分、巡检流水线、FDR、校准全是后台**——用户永远不该看见"印证系数"这种词。他只看见"XX医院在变虚，因为这三件事，建议两天内回访"。这把产品从"要人主动想、主动查"的分析工具，变成"线索自己找上门、处理完打个勾"的工作流。

### 8.3 买家 = 营销 VP = 日活 = 采纳强制器（含陷阱）

- 买单的是**企业营销部门 VP，且每天登录 SaaS**。买家=日活，在 to-B 里是稀缺的好事。
- **关键牌**：VP 在驾驶舱里看见处置情况，压力自上而下传，把采纳变成管理指令——不靠一线自发热爱。（注：此牌在 v1 休眠，见 8.6。）
- **陷阱**：自上而下的压力会催生一线刷"已处置/已挽回"造假，污染算法标签。解药见 8.4。

### 8.4 兑现率 KPI + 线索状态机（抗造假核心）

进驾驶舱的核心 KPI 是「**挽回兑现率**」，**不是处置率/响应率**。考核处置率→一线刷处置；考核兑现率→刷不动，因为订单数据他改不了。**让数据当裁判。**

落地为挂在每张卡上的**线索状态机**（不另建臃肿工单系统——现 SaaS 也无工单可接管）：

```
待处理 → 跟进中 → 已处置(声称挽回 / 确认流失 / 误报+原因)
                          ↓ 系统下个巡检周期自动回填核验
                   兑现 / 未兑现(声称挽回但订单没回来 → 自动翻红重开，标记"声称挽回未兑现")
```

线索卡本身就是工作单元，它的生命周期就是工作流。这套反馈同时三用途：给操作者信任、给算法喂干净标签（附录 C 校准/uplift 的燃料）、采集"干预-结果"配对。

### 8.5 隐形 agent（v1 不做聊天框）

v1 的智能体现在"主动生成会说人话的卡 + 不用被问就来找你"，**不是"问它答"**。一线/VP 没有高频查数需求，需求是 push 不是 query。聊天式"问问我的数据"是 v2 锦上添花，不是切入点。

### 8.6 v1 关键约束与收缩：无组织数据 → 中央收件箱（VP 本人操作）

> 这是产品形态最硬的一条现实约束，逼出一个诚实的 v1 收缩。

**约束**：客户**没有任何销售组织/区域划分数据**——订单只有 医院/药品/配送商，系统不知道一线是谁、终端归谁管。**v1 无法自动路由、无法分布式推送到一线兜里**（8.1 的"逃逸"在 v1 受限）。

**收缩**：v1 的用户就是唯一确定存在且每天登录的人——**VP 本人**。形态坍缩成**一块管理层屏幕**：

```
┌─ VP 每日屏（驾驶舱与收件箱合一，5 分钟）──────────┐
│ 顶部：风险敞口 · 兑现率 · 挽回 ROI（证明，给买家看） │
│ 下部：按【挽回价值】排序的 Top N 条可动作线索        │
│       每条 = 证据链 + 建议动作 + 一键"转给[某人]"     │
└──────────────────────────────────────────────┘
```

- **管理层轻量**：只把 红 + 头部 橙 浮到 VP 眼前，黄收进观察清单。VP 时间稀缺，过载即弃用——防疲劳在这里是最高优先级。
- **一键企微转发 = v1 的逃逸 + 组织图累积器**：系统把卡片话写好，VP 只挑收件人、点发送，线索进入某人企微。每转一次，顺手记一笔「终端→人」。这一个动作同时完成"逃逸"和"累积客户自己都没有的销售组织图"——**弱点变飞轮、变粘性**。
- **v1 近乎单用户产品**：无需角色权限、团队协作、路由引擎、经理看板。范围小到能快速做出、且数据撑得住。

### 8.7 三层产品面与解锁次序

| 触点 | 谁 | 看什么 | 在哪 | 阶段 |
|---|---|---|---|---|
| 每日屏（驾驶舱+收件箱合一） | VP 本人 | 敞口→覆盖→兑现率→ROI + Top N 线索 | SaaS 内 | **v1** |
| 派单看板 | 区域经理 | 队列、路由、辖区覆盖 | SaaS 内 | v2 |
| 线索收件箱 | 一线业务员 | 卡片+证据+动作+一键反馈 | 企微逃逸 | v2 |

v2 的经理/一线层，靠两条解锁：① 试点只需 VP 手工填**一个辖区**的微映射，即可在该辖区开企微推送跑通整环；② 中央分派累积出的「终端→人」映射够密后，系统级自动推送成立。届时 8.3 的"采纳强制器"才激活。

### 8.8 上线姿势：回测 → 影子 → 试点 → 铺开（决定开发次序）

预测类产品第一印象一旦错，买家永久打折。所以不用"实时预测"姿势上线，用"回测证明"姿势上线，分四阶，信任一阶比一阶厚：

1. **回测证明**：喂客户去年真实流失终端，站在他已知答案的数据上证明"提前 N 天亮灯+附证据"。第一个产品形态几乎就是系统自动生成的一份咨询级回测报告，也是签约弹药（=第十一章 The Assignment / 附录 C.6）。
2. **影子运行**：活体巡检但不推一线，和 VP 一起在这家客户数据上调阈值（VP 共同背书）。
3. **试点逃逸**：手工微映射一个辖区，开企微推送，跑通整环、量兑现率。
4. **铺开**：扩辖区，再复制其他客户。

**改变开发次序**：先建回测引擎 + 影子复盘界面，再建精致的企微推送——别拿生模型怼一线，第一个误报就烧掉买家。

### 8.9 v1 风险与护栏

- **单用户集中风险**：v1 押在一个 VP 身上，他一忙就没人撑着这产品。护栏：用回测 ROI / 兑现率持续证明省了多少钱，让 VP 有理由每天来；并有紧迫感尽快推进到**试点逃逸**，加入真正每天用的一线用户，别长期停在中央模式。
- **证据链是 v1 的唯一拐杖**：中央操作的 VP 没有一线对每家医院的现场感，完全靠证据链读懂每条线索。证据链质量从"采纳前提"升级成"v1 能不能用"。
- **防过载**：宁可少报，红橙才进 VP 视野，黄入观察清单——参见 6.6 分级与 6.9 挽回价值排序，它们在这里直接决定 VP 弃不弃用。

## 九、开放问题（Open Questions）

1. **判定单元**：「医院×单药品」还是「医院×产品线」为风险分最小单元？需与客户业务口径对齐。
2. **主干假设边界**：BG/NBD 的非契约/同质假设在医院采购上偏差多大？政策性流失（集采落标）与自然流失能否在数据上清晰切分。
3. **基线冷启动**：低频采购药、新合作医院历史样本不足时的同侪先验如何标定。
4. **正常波动剔除**：集采周期、挂网状态变化、节假日如何在 L2 扣除。
5. **同侪对照可行性**：是否有足够同等级/同地区医院样本支撑方向7。
6. **终端价值数据**：能否从历史采购金额 + 机构等级估出可用的终端价值/CLV 分层（决策排序的前提）。
7. **可挽回性数据（uplift）**：需积累多少"干预-结果"配对，才够训练 uplift 模型？v1 先靠工单回流采集。
8. **现状对照基线**：首访量化客户当前"从风险苗头到察觉"的平均耗时。

## 十、成功标准（Success Criteria）

- **提前量命中**：用客户真实流失终端回测，在真正停购前给出橙色及以上预警的比例（如提前 ≥30 天命中 ≥60%），经 walk-forward CV 验证。
- **概率校准**：风险分与实际流失率的校准误差（可靠性图/Brier 分）达标——分数要"名副其实"。
- **误报可控（FDR）**：经 Benjamini-Hochberg 控制后，红/橙预警的一线"无效"反馈比例低于阈值。
- **决策有效性**：高"挽回价值"线索的实际挽回成功率显著高于随机派单（验证价值加权/uplift 排序确实有用，而非只是预测准）。
- **一线真实使用**：核心用户每周主动查看/处置风险线索的留存。
- **挽回闭环**：经预警触发并最终"风险解除/已挽回"的终端数 > 0，且客户认可这是工具的功劳。
- **买单转化**：已表达买单意愿的客户由"意愿"转为"签约/付费"。

# 附录 A · 探测器参数默认值与伪代码

> 约定：分析单元 u = (医院编码 h, 产品线 p)；下钻单元 = (h, 品规 sku)。所有"默认值"为**起步值**，须经附录 C/回测校准。`s` = 严重度∈[0,100]，`c` = 置信度∈[0,1]。

## A.0 公共预处理（L0）

```
对每个分析单元 u：
  orders_u = 按下单时间排序的订单序列（按 (h,p) 聚合，sku 级保留供下钻）
  IF 历史订单数 n_u < N_MIN: 标记 cold_start=True
  # 去伪用的同群（L2/方向7 复用）
  cohort(u) = 同 机构等级 + 同 所属地区 的医院集合
  # 去季节：同比同期基线
  deseason(x_t) = x_t / median(同群同月历史值)   # 简化版，数据足时换 STL
默认：N_MIN = 6（单元至少 6 单才建独立基线，否则走冷启动先验）
       窗口：recent = 近 90 天；base = 近 91–365 天
```

## A.1 主干 P(alive)（BG/NBD）

```
# 输入：每个医院 h 的 RFM —— T(首购距今), t_x(末购距今 in 校准期), x(重复购买次数)
fit BG/NBD on 全体医院 → 参数 (r, alpha, a, b)        # 季度重拟合一次
for h:
  P_alive(h) = BGNBD_palive(r,alpha,a,b | x, t_x, T)
backbone_score(u) = (1 − P_alive(h)) × 100
# 工程兜底：BG/NBD 拟合不稳时，用 RFM 经验分位的可解释代理：
#   P_alive_proxy = 1 − ECDF_recency( 距上次下单天数 | 同采购形态簇 )
默认：重拟合周期 = 季度；P_alive 走势取近 3 个巡检点斜率作"下行"判定
```

## A.2 探测器1 · 采购节奏异常（IPI）

```
ipi = 相邻订单下单时间差序列
med = median(ipi);  mad = 1.4826 × median(|ipi − med|)   # 稳健
dslo = 今天 − 末次下单时间
z = (dslo − med) / max(mad, MAD_FLOOR)
hit  = z ≥ Z_HIT
s    = clip( 100 × (z − Z_HIT)/(Z_FULL − Z_HIT), 0, 100 )
c    = clip( n_u / N_CONF, 0, 1 )
默认：Z_HIT = 1.5（≈80 分位），Z_FULL = 3.5，MAD_FLOOR = 3 天，N_CONF = 12
注：cold_start 时 med/mad 用同采购形态簇先验，c 上限 0.5
```

## A.3 探测器2 · 量/金额趋势衰减

```
series = 近 12 个月 月采购数量（去季节后）
tau, p = MannKendall(series)            # tau<0 且 p<MK_P 视为显著下行
slope  = TheilSen(series)               # 盒/月 的下滑斜率
# CUSUM 抓持续性下移
target = mean(base 窗均值); sd = std(base)
C⁻ = running max(0, C⁻_prev + (target − k·sd − x_t))
hit  = (tau < 0 AND p < MK_P) OR (C⁻ ≥ H·sd)
drop = max(0, −slope × 3 / target)       # 近 3 月相对下滑
s    = clip( 100 × drop / DROP_FULL, 0, 100 )
c    = clip( 有效月数 / 12, 0, 1 )
默认：MK_P = 0.1，CUSUM k = 0.5、H = 4（即 4σ），DROP_FULL = 0.6（跌 60% 记满分）
```

## A.4 探测器3 · 采购频次衰减

```
λ_base   = base 窗内 月均下单次数
λ_recent = recent 窗内 月均下单次数
# 泊松率下降显著性（比率检验 / 简化用泊松尾概率）
rate_ratio = λ_recent / max(λ_base, EPS)
hit = rate_ratio ≤ FREQ_DROP AND λ_base ≥ FREQ_MIN_BASE
s   = clip( 100 × (1 − rate_ratio)/(1 − FREQ_DROP), 0, 100 )
c   = clip( λ_base / 2, 0, 1 )           # 本就低频的，频次维度可信度低
默认：FREQ_DROP = 0.6（频次降到 6 成以下），FREQ_MIN_BASE = 1（月均≥1 单才用此维度）
```

## A.5 探测器4 · 品规收窄 / 钱包份额

```
sku_base   = base 窗内 活跃品规数（distinct 药品编码/规格）
sku_recent = recent 窗内 活跃品规数
shrink = max(0, (sku_base − sku_recent) / max(sku_base, 1))
hit = shrink ≥ SKU_SHRINK AND sku_base ≥ SKU_MIN_BASE
s   = clip( 100 × shrink, 0, 100 )
c   = clip( sku_base / 3, 0, 1 )          # 本就单品规的医院，此维度不适用
默认：SKU_SHRINK = 0.34（≥1/3 品规消失），SKU_MIN_BASE = 2
价值：品规逐个掉 = 竞品逐步替代，常**领先于**总量下滑，是最早信号
```

## A.6 融合与分级（L3/L4，对照 6.5/6.6）

```
families = {F1=IPI, F2=趋势, F3=频次, F4=品规, F5=履约(见附录B), F6=价格, F7=同侪}
k = Σ 1[ family.hit AND family.c ≥ C_MIN ]      # 独立命中信号族数
# 同侪去伪（方向7）：若本院相对同群非特异，整体压制
suppress = (相对同群下滑分位 < PEER_PCTL)
corr = min( 1 + ALPHA·(k − 1), CORR_CAP )
Score = clip( backbone_score × corr × (SUPPRESS_F if suppress else 1), 0, 100 )

# 分级（印证逻辑，非纯阈值）
IF P_alive 低 AND k ≥ 2 AND not policy_masked AND not suppress:   级别 = 红
ELIF (P_alive 中) OR (k = 1 强信号族 AND 同侪确认特异):           级别 = 橙
ELIF k ≥ 1:                                                      级别 = 黄
ELSE:                                                            不预警
默认：ALPHA = 0.5，CORR_CAP = 2.0，C_MIN = 0.4，PEER_PCTL = 50%，SUPPRESS_F = 0.4
     P_alive 低 < 0.4、中 0.4–0.7（阈值回测校准）
```

> 实现顺序建议：先 A.2/A.5（趋势+品规，最易算、信号最实）跑通闭环，再接 A.1 IPI 与 A.1 主干 P(alive)，最后融合。

# 附录 B · 履约三段差专项（探测器5，医药独有的金矿）

## B.1 为什么单列

订单天然分三段：**采购数量（医院想要）→ 配送数量（配送商发的）→ 收货数量（实际到的）**。三段之间的缺口，能把一件事拆开：**"医院不想买了（需求流失）" vs "医院想买但买不顺（断供型流失）"**。两者根因和干预动作完全不同——前者找医院做客情，后者换/催配送商。混判会让一线跑错方向，是最贵的误报。

## B.2 订单级履约指标

```
对每条订单 o：
  配送率 fill_o   = 配送数量 / max(采购数量, EPS)
  收货率 recv_o   = 收货数量 / max(配送数量, EPS)
  整体履约 otif_o = 收货数量 / max(采购数量, EPS)
  配送时延 dlag_o = 配送时间 − 下单时间        # 天
  收货时延 rlag_o = 收货时间 − 配送时间
  拒单 reject_o   = 1[ 配送数量 = 0 OR 超 REJECT_DAYS 天无配送记录 ]
默认：REJECT_DAYS = 7
缺口归因：
  采购-配送缺口 (采购数量 − 配送数量) > 0  → 配送商未足额发货（断供）
  配送-收货缺口 (配送数量 − 收货数量) > 0  → 退货/验收/质量问题
```

## B.3 聚合到 (医院×产品线×窗口) 的趋势

```
窗口取近 90 天 vs 历史：
  fill_rate_trend  = CUSUM/MK 于 月度 mean(fill_o)        # 配送率下行
  dlag_trend       = MK 于 月度 median(dlag_o)            # 时效拉长
  reject_rate      = mean(reject_o) over recent
hit_fulfillment = (fill 趋势显著下行) OR (dlag 显著拉长) OR (reject_rate ≥ REJECT_RATE_HIT)
s = 加权 clip 组合(配送率跌幅, 时延增幅, 拒单率)
c = clip( recent 订单数 / N_CONF, 0, 1 )
默认：REJECT_RATE_HIT = 0.15，配送率"健康线" FILL_OK = 0.8（呼应需求文档配送率<80%预警）
```

## B.4 断供型 vs 需求型 判别（B 的核心产出）

履约探测器除了贡献风险分，还输出一个**根因标签**，用来给整条线索定性、重路由处置：

```
purchase_healthy = NOT 探测器2.hit AND NOT 探测器3.hit     # 采购侧没明显下滑
fulfillment_bad  = hit_fulfillment

IF purchase_healthy AND fulfillment_bad:
    标签 = "断供型风险"        # 医院在正常下单，但供不上 → 客户≠根因
    建议动作 = 核查/更换配送企业，催配送
    风险归属 = 配送商，不计入"医院需求流失"主因（避免冤枉客户）
ELIF (NOT purchase_healthy) AND fulfillment_bad:
    标签 = "需求下滑 + 履约恶化叠加"   # 最危险，双重流失压力
ELIF (NOT purchase_healthy) AND NOT fulfillment_bad:
    标签 = "纯需求型风险"      # 供应正常，医院就是在少买 → 找医院
ELSE:
    标签 = "暂无履约异常"
```

## B.5 配送商级系统性检测（防止冤枉医院）

```
按 配送企业 d 聚合：
  d_fill   = mean(fill_o | 配送企业=d, recent)
  d_scope  = distinct 医院数(fill_o<FILL_OK | d)
IF d_fill 显著低 AND d_scope ≥ SYS_MIN_HOSP:
    生成独立线索类型 = "配送商系统性失约"（修配送商）
    并对受影响医院的"流失风险分"做**履约因子抵扣**——它们的低配送是系统性的，不是各自的需求问题
默认：SYS_MIN_HOSP = 5
```

## B.6 证据链片段（履约型）

```
风险线索：YY二甲医院 · 本企业B产品线 · 风险分 71 · 标签【断供型风险】
履约三段差：
  · 配送率   近3月 0.93→0.78→0.61（健康线 0.80）   MK 显著下行
  · 配送时延 中位 2→6→11 天                         明显拉长
  · 拒单率   近90天 18%（3 单 0 配送）
采购侧：采购数量基本平稳（医院仍在正常下单）
判别：采购健康 + 履约恶化 → 断供型，根因指向配送企业【ZZ医药】
建议动作：商务联系 ZZ医药核查发货能力，必要时切换配送渠道；勿按需求流失处置
```

> 落地次序：B.2 订单级指标 → B.3 趋势 → B.4 判别标签（这步价值最高，先做）→ B.5 配送商级（数据量够了再上）。

# 附录 C · 回测协议（Hyndman + Provost 视角 · 权重阈值的唯一裁判）

> 6.5(3) 与第十章引用本附录。**没有这套协议，所有默认值都是拍脑袋，分级阈值无意义。** 第十一章「拿客户去年真实流失终端做离线回测」即是本协议的首次执行。

## C.0 为什么要单列一套协议

三个最容易在医药时序数据上翻车的坑，附录 A/B 的默认值都救不了，只能靠回测裁判：

1. **时间穿越**：用未来信息算过去的特征 → 线下虚高、线上崩盘。
2. **多重检验假阳**：每天对数千单元跑多探测器，纯随机也能"发现"成百上千条假风险。
3. **分数名不副实**：风险分 78 若不对应约 78% 的真实流失率，红/橙/黄阈值就是空中楼阁。

C.1 治穿越，C.3 治假阳，C.4 治校准。C.2 量提前量（产品的价值证明），C.5 量"挽回"（决策有没有用）。

## C.1 Walk-forward 时序交叉验证（禁随机 k 折）

```
绝不用随机 k 折——会把未来订单混进训练，时间穿越。
改用滚动原点（rolling-origin）前向验证：

  把历史按月切成 t0 < t1 < ... < tN
  for 每个评估原点 ti （从有足够历史的最早点起）:
      训练窗 = [起点, ti]            # 只用 ti 之前的数据拟合 BG/NBD、定阈值
      验证窗 = (ti, ti + H]          # H = 提前量目标，默认 90 天
      在 ti 这天对全量单元跑完整流水线 L0..L4，记录每个单元的预警等级
      到 ti+H，揭晓哪些单元真停购，比对预警
  指标在所有 ti 上聚合（macro 平均），看跨期稳定性，不是单点

关键纪律：
  · 特征只能用 ti 及之前的订单（含基线、同群、季节因子全部 as-of ti）
  · BG/NBD 参数、各探测器阈值、FDR 临界都在训练窗内定，验证窗只评估不调参
  · 调参用更早的内层窗（nested），别拿验证窗反复试——那也是穿越的一种
```

## C.2 提前量与命中（产品的核心价值证明）

```
先定"真流失"金标准（与客户业务口径对齐，见开放问题1）：
  churn_true(u) = 该单元在某时点后连续 ≥ L_STOP 天无有效采购，且客户确认非集采落标/正常停用
  默认 L_STOP = 历史中位采购间隔 × 3，下限 90 天

对每个真流失单元 u，回看它"真正停购日 d_stop"之前：
  lead_orange(u) = d_stop − 首次橙及以上预警日     # 提前多少天进入"该跟进"
  lead_red(u)    = d_stop − 首次红预警日

汇报三条曲线（一线和客户都看得懂）：
  · 命中率@提前量：在 d_stop 前 ≥30/60/90 天给出橙+ 预警的真流失占比
  · 提前量分布：lead_orange 的中位与四分位（不是只报均值，长尾很重要）
  · 等级纯度：红预警里真流失占比（precision）、真流失里曾被红覆盖占比（recall）

成功标准锚点（第十章）：提前 ≥30 天命中 ≥60%，经 walk-forward 跨期复现，非单期碰运气。
```

## C.3 FDR 多重检验控制（Benjamini-Hochberg · 回测期同样执行）

```
每个评估原点 ti，把当天所有 (单元 × 探测器) 的显著性检验 p 值收齐：
  p_(1) ≤ p_(2) ≤ ... ≤ p_(m)            # 升序
  找最大 k 使 p_(k) ≤ (k/m)·q            # q = 目标 FDR，默认 0.10
  拒绝前 k 个原假设 → 这些命中才算"经 FDR 存活的独立信号族"
  只有存活命中才进入 6.5 印证升级、6.6 分级

回测要报：
  · FDR 实际控制水平（验证窗里"红/橙但客户判无效"的比例 ≈ 经验 FDR）
  · 开/关 BH 的 A/B 对照：证明 BH 把假阳压下来了，且没把真阳一起杀光
注：间断/块状单元（L0.5）本就不跑 MK 等趋势检验，不进入这批 p 值池。
```

## C.4 概率校准（Platt / Isotonic · 分数要名副其实）

```
判别力（AUC/排序对不对）≠ 校准（分数等于真实概率）。本产品分级靠绝对分，必须校准。

在验证窗上：
  把单元按风险分（或模型输出的流失概率）分箱（如 10 个等频箱）
  每箱画 (平均预测概率, 实际流失率) → 可靠性图（reliability diagram）
  理想是 45° 对角线；系统性偏离就需校准

  校准方法：训练窗上拟合 Platt（逻辑回归压缩）或 Isotonic（单调回归），
            作用在原始分→校准概率，验证窗只评估
  指标：Brier 分（越低越好）、ECE 期望校准误差、可靠性图目检

  分级阈值（红<P 中 0.4–0.7 等）必须定义在【校准后概率】上，不是原始判别分——
  否则 6.6 的 0.4/0.7 没有统计含义。
```

## C.5 Uplift / 挽回价值评估（决策有没有用，不只是预测准）

```
6.9 的命题：一线该按"挽回价值"派单，不是按风险分。回测要能验证这条真成立。

v1（还没有干预-结果配对时）——先验证价值加权：
  对已处置线索，按 风险分×价值档 分层，比较各层"实际挽回成功率"
  目标：高价值层挽回率显著高于随机派单基线（证明别把人力砸在挽不回的小户上）

v2（积累足够"干预-结果"配对后）——真 uplift 评估：
  需要干预的随机性来源：理想是小样本随机派单/不派单对照；
  否则用工单回流的自然变异 + 倾向得分校正（次优，标注偏差风险）
  评估用 Qini 曲线 / uplift@k：被模型选中投入干预的 persuadables，
  其留存提升是否真高于随机选取同量终端
  护栏（Provost）：低置信/低校准线索不强行编 uplift，显式标"待积累数据"
```

## C.6 回测最小执行清单（对应第十一章 The Assignment）

```
1. 找客户要 5–10 个去年真实流失终端 + 同群对照的正常终端（防只看阳性）
2. 拉停购前 12 个月订单（不止 6 个月——base 窗需要 91–365 天）
3. 对每个终端，按 C.1 设 d_stop，回放流水线，记录 lead_orange / lead_red
4. 跑 C.2 三条曲线；跑 C.3 开关 BH 对照；跑 C.4 可靠性图
5. 把"哪几个维度最早亮、客户最信哪个"带回来 → 校准 ALPHA / 各探测器阈值 / 分级线
6. 产出一页《回测报告》：提前量中位、命中率@30天、经验FDR、Brier分——这页就是付费转化的弹药
```

> 一句话：附录 A/B 给的是**起步默认值**，附录 C 是**把默认值变成事实的唯一通道**。先做 C.6 清单，再谈上线。

# 附录 D · 数据接入口径（工程开工前最后一块拼图 · 与附录 C 回测取数共用）

> 对应 8.1「数据层接入」。产品独立、数据只读接老 SaaS 底层订单库。本附录把"接数据源就可以"落成可执行的取数口径，**同一份口径既喂日常巡检、也喂附录 C 回测**。

## D.0 接入原则

- **只读，不回写**老系统；产品自己的状态机/兑现核验数据存在独立产品库里。
- **中间隔一层标准字段 schema（规范层）**：底层是直连只读副本还是视图，是 v1 实现细节；算法只认规范层字段。否则铺开到 200 家、每家库结构不同时要重写。
- **日频批量增量 + 滚动回填窗**：巡检日频，每日增量同步即可，无需实时。**但订单是"活"记录**——下单后配送/收货字段隔天甚至隔周才回填，增量不能只按"新订单号"，须用变更时间戳（CDC）或**每日重拉滚动窗（如近 90 天）**捕获回填。
- **多租户**：每家药企只采本企业产品订单，数据按生产企业天然隔离；v1 单客户，架构先留隔离口子。

## D.3 四个工程口子（钉死再开工）

1. **历史深度**：日常 base 窗需 91–365 天；**回测要 walk-forward 多原点**，至少 12 月 base + 数月滚动原点 + 验证窗 → **实需 ≥18–24 个月历史**。须确认底层库实际存了多少（见 D.4⑤）。
2. **增量与回填**：见 D.0，配送/收货滞后回填是履约探测器(B)的命门，回填窗不够会把"还在途"误判成"拒单"。
3. **订单粒度与唯一键**：一个订单编号一行，还是分次配送多行？决定聚合逻辑（见 D.4③）。
4. **多租户隔离**：按生产企业。

## D.4 必须向数据方/客户确认的缺口清单

① **产品线如何定义**：源数据只有 SKU（药品编码/通用名/规格/批文），没有"产品线"字段。SKU→产品线映射从哪来（按通用名自动卷？还是客户给一份产品主数据）？——分析单元的前提。
② **"本企业产品"口径**：靠生产企业字段还是批准文号主数据界定？字段干净度如何？
③ **订单编号是否唯一 / 分次配送是否多行**：影响频次、履约的聚合。
④ **配送/收货字段回填时序**：下单后平均多久有配送、多久有收货数据？——直接定 REJECT_DAYS 与履约探测器可信度。
⑤ **实际历史深度**：底层库到底有多少个月可回溯？不足 18 个月，walk-forward 回测要降级设计。
⑥ **集采/挂网状态可得性**：L2 去伪要识别集采落标。医保目录类型够不够，还是需补挂网/中标状态外部源？

## D.5 回测取数 = D 全量 + 三件回测专属（接附录 C）

附录 C.6 的离线回测，取数就走本附录 D，额外三件：
- **标签**：客户标出 5–10 个去年真实流失终端的 (医疗机构编码, 产品线) + d_stop 大致时点；**另要一批同群"正常"终端做对照**，否则只看阳性、命中率虚高。
