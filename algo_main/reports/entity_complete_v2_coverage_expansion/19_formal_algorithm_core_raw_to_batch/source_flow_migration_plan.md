# Source Flow Migration Plan

## Goal

Migrate the verified v2 exploration flow into `risk_algorithm_core` without importing `algo_main` at runtime.

## Runtime Input Decision

The current repository contains `fact_purchase_event.parquet`, which is a cleaned order fact generated by the verified v2 flow. The current formal runtime therefore uses `normalized_fact_mode` as the first auditable mode.

`formal_raw_orders_mode` remains the production target once raw order tables are provided through the standard raw input batch contract.

## Functions Migrated Into risk_algorithm_core

| Stage | Source | Runtime Module | Runtime Function | Notes |
|---|---|---|---|---|
| fact purchase event | `alg.facts.purchase_event_builder` | `risk_algorithm_core/facts.py` | `build_fact_purchase_event_from_orders` | Migrated logic; accepts standard raw orders or normalized fact-compatible orders. |
| entity month | `alg.facts.entity_month_builder` | `risk_algorithm_core/entity_month.py` | `build_fact_entity_month` | Migrated aggregation semantics. |
| purchase sequence | `entity_complete_rebuild.py` | `risk_algorithm_core/purchase_sequence.py` | `build_entity_purchase_sequence` | Migrated sequence/gap logic. |
| cutoff features | `entity_complete_rebuild.py` | `risk_algorithm_core/cutoff_features.py` | `build_source_cutoff_features`, `build_features_for_cutoff` | Migrated source feature logic. |
| baseline model fields | `entity_complete_rebuild.py` | `risk_algorithm_core/cutoff_features.py` | `add_baseline_scores` | Produces `recency_only_baseline`, `frequency_decay_baseline`, `interval_overdue_baseline`, and `hybrid_interval_frequency_score`. |
| model feature alignment | artifact schema | `risk_algorithm_core/production_feature_builder.py` | `build_model_feature_frame` | Uses artifact feature order; existing NaN values are left for training pipeline imputers. |
| candidate policy | `entity_complete_algorithm_consolidation.py` / `entity_complete_v2_coverage_expansion.py` | `risk_algorithm_core/candidate_selector.py` | `BoundedCandidateSelector`, `candidate_policy_members` | Migrates `multi_recall_union_top10`; per-user worklist fill is not core M1. |

## Hard Blockers

- Missing raw order source blocks `raw_orders_mode_ready`.
- Fact parity failure blocks `conditional_fact_mode_ready`.
- Feature parity failure blocks formal second-layer readiness.
- Score parity failure blocks formal second-layer readiness.
- Result batch cannot be declared final if M-stage parity is not proven.

## Per-User 20-50 Worklist Rule

The 20-50 requirement is a presentation/workload rule. It must be applied after core M1 candidate selection using user/org scope. It cannot replace `multi_recall_union_top10` or any verified M1 source policy.
